# 구구단 스퍼트(multiplication-sprint) 구현 노트 (2026-07-16)

텐텐오락실 오마주 후보 10종(Phase 4) 중 네 번째 게임. 업다운 넘버까지는 "숨은 정답 하나를
계속 추론"하는 구조였다면, 이 게임은 "짧은 문제가 반복적으로 갱신되는" 구조라 처음으로
다른 뼈대가 필요했다.

## 스펙에 없어 직접 정한 수치와 근거

- **2~9단, 문제 간격 3초**: 표준 구구단 범위. 암산 스퍼트라는 컨셉상 문제를 오래 띄워두면
  긴장감이 없어서, 안 풀어도 3초 뒤 강제로 다음 문제로 넘어가게 했다.
- **라운드 45초(약 15문제)**: 두더지 사냥(30초)보다 살짝 길게 잡았다 — 문제 15개 정도는
  풀어봐야 "스퍼트"라는 느낌이 나고, 15문제 동안 순위가 여러 번 뒤집힐 여지가 생긴다.
- **랭킹 = 정답 개수 내림차순, 동률이면 평균 응답속도**: 스펙 문구("제한시간 내 연속
  정답 수·속도로 실시간 랭킹")의 "연속"을 "끊기지 않고 맞추는 콤보 스트릭"이 아니라
  "연속으로 제시되는 문제들에 대한 누적 정답 수"로 해석했다 — 콤보 스트릭 방식은 한 번
  실수하면 그동안 쌓은 게 전부 날아가 버려 캐주얼한 파티게임 톤에 비해 지나치게 가혹하다고
  판단했다.
- **문제당 첫 정답만 인정**: 같은 문제를 여러 번 다시 제출해도 정답 개수가 중복으로
  올라가지 않도록 `answeredThisProblem` 플래그로 막았다. 오답은 재시도 제한 없이 허용
  (정답을 못 맞혀서 압박을 느끼는 게 아니라 속도로 승부하는 게임이라 오답 페널티는
  불필요하다고 판단).

## 서버 권위 원칙이 단순해지는 지점

업다운 넘버는 정답 숫자가 비밀이라 힌트만 사설 전송하고 다른 사람에게는 시도 횟수만
브로드캐스트해야 했지만, 이 게임은 `a×b` 문제 자체가 애초에 전원에게 공개되는 값이라
숨길 게 없다. 그래서 `problem` 메시지를 전원에게 그대로 브로드캐스트해도 안전하고,
`progress_update`에도 별다른 익명화 없이 정답 개수만 넣으면 된다 — 안티치트 설계가
업다운 넘버보다 단순해졌다는 걸 라이브 프레임 검사로 재확인했다.

## 검증

- **raw WebSocket 스크립트**: 호스트+게스트 2개 소켓으로 방 생성→시작→카운트다운→라운드
  시작 후, 서버가 보낸 `a×b` 문제를 그대로 계산해 5문제 연속 정답을 제출하며 `correctCount`가
  1부터 5까지 순서대로 올라가는지 확인. 게스트가 고의 오답을 제출해도 `correct:false`만
  오고 크래시나 카운트 오염이 없음을 확인.
- **Playwright UI 테스트**: 모바일 세로(390×844)에서 로비 → 카운트다운 → 플레이 화면까지
  실제 UI로 문제 텍스트를 파싱해 정답을 입력하고, "정답!" 초록 플래시와 보드의 정답 개수
  갱신을 스크린샷으로 확인.
