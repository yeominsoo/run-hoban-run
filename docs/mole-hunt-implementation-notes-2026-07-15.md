# 두더지 사냥(mole-hunt) 구현 노트 (2026-07-15)

텐텐오락실 오마주 후보 10종(Phase 4) 중 첫 번째 게임. 계획서(`docs/game-collection-plan-v2-
2026-07-13.md`)의 "동시 랭킹형" 패턴을 이 게임에서 처음 확립했고, 나머지 6종(순서 기억
챌린지·업다운 넘버·구구단 스퍼트·홀짝 암산·컬러 지시 게임·합이 10 퍼즐)은 문제 생성 로직만
바꿔 이 구조를 재사용하면 된다.

## 확정한 수치 (텐텐과 무관하게 자체 정의 — 오마주 안전장치 준수)

- 방 인원: 2~8명 (반응속도 게임이라 인원이 많아도 서버 부하가 적음)
- 구멍: 3×3(9개), 라운드 시작 카운트다운 3초, 라운드 길이 30초
- 두더지 노출 시간 850ms, 다음 두더지까지 550~950ms 랜덤 대기 → 30초에 평균 35~40마리
- **동시에 하나의 두더지만 활성화**(할리갈리의 "하나의 종치기 대상" 구조와 동일한 결정) —
  여러 명이 각자 화면에서 같은 두더지를 노리는 경쟁 구도를 만들기 위해서였고, 구멍마다
  독립적으로 여러 마리가 동시에 뜨는 방식보다 "누가 먼저 탭했는가"를 서버가 명확히 판정하기
  쉬워 안티치트/재현성 면에서도 유리하다.

## 서버 권위 판정

할리갈리의 "서버 수신 순서 기반 판정" 원칙을 그대로 따른다. 두더지가 뜨면 `mole_spawn`을
전원에게 브로드캐스트하고, 각 클라이언트가 보낸 `hit_mole`은 Node 이벤트 루프가 처리하는
순서 그대로 "첫 번째 유효한 히트"만 인정한다(`room.activeMole.resolved` 플래그로 이중 판정
방지). 노출 시간 내에 아무도 못 맞히면 `hideTimer`가 만료돼 miss로 처리하고 바로 다음
두더지를 스케줄링한다.

## 점수/랭킹

라운드 종료 시 최고 점수를 받은 전원을 승, 나머지를 패로 `ranking-store.mjs`(주간 파일
기반, 다른 5개 파티게임과 동일 패턴)에 기록한다. 전원 0점(아무도 못 맞힘)이면 랭킹에 아예
기록하지 않는다 — 의미 없는 전원 패배 기록을 남기지 않기 위함. 게임 도중 인원이 2명 미만으로
줄면(이탈) 점수와 무관하게 남은 한 명을 즉시 승자로 기록하고 종료한다(다른 파티게임들의
"조기 이탈 시 강제 종료" 정책과 동일).

## 검증

로컬(`PORT=8799`)에서 raw WebSocket 스크립트로 3가지 시나리오를 실제로 플레이해 확인:
1. 정상 플레이 — 30초 라운드 동안 38마리 스폰, 반응이 30ms 빠른 클라이언트가 38:0으로
   전승(서버 수신 순서 판정이 실제로 "빠른 쪽"에게 유리하게 작동함을 확인).
2. 게임 도중 한 명이 `leave` → 남은 플레이어 즉시 승리(`opponentLeft: true`), 점수 무관하게
   랭킹에 승리로 기록됨.
3. 소켓만 끊고(`ws.close()`) `rejoin` → 진행 중이던 라운드의 점수판과 활성 두더지 상태를
   그대로 이어받음.
4. `GET /ranking/mole-hunt` 응답에 방금 기록한 승/패가 즉시 반영됨을 확인(파일 저장은
   2초 디바운스지만 조회는 인메모리 상태를 바로 읽으므로 지연 없음).

WAS 배포는 다른 5개 파티게임과 동일한 절차(`ws-server/README.md`의 재배포 절차)를 그대로
따랐다 — rsync → `docker build` → `rps-server` 컨테이너 교체(같은 named volume 재사용) →
`~/rps-tls/nginx.conf`에 `/mole-hunt` location 추가 → `docker restart rps-tls`.
