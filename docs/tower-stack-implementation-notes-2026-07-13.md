# 타워 쌓기 구현 노트 (2026-07-13)

## 원본 스펙 (v1)

- 왔다갔다 움직이는 블록을 탭해서 정확하게 쌓기
- 입력: 탭 1회
- 규칙: 이전 블록과 겹치는 부분만 남음, 삐진 부분 잘림
- 종료 조건: 블록 너비가 10px 미만이면 게임 오버
- 점수: 층 수

이 장르는 Ketchapp의 "Stack" 류 게임과 동일한 잘 알려진 구조라, 그 장르의 표준 규칙을
그대로 따르되 구체 수치만 새로 정했다.

## 결정과 이유

1. **난이도 곡선(이동 속도)**: **시간이 지나도 이동 속도를 올리지 않는다(상수 속도).** v1
   스펙에 속도 증가에 대한 언급이 전혀 없다(반면 볼 피하기·에임 트레이너는 v1에 명시적인
   속도/크기 증가 규칙이 있었다). 이 게임은 "겹치는 부분만 남고 잘림" 규칙 자체가 이미
   자기강화적 난이도 곡선이다 — 살짝만 삐끗해도 블록이 좁아지고, 좁아진 블록은 같은 이동
   속도에서도 허용 오차가 훨씬 작아져 자연스럽게 어려워진다. 임의의 속도 증가 수치를 새로
   지어내지 않았다.
2. **미스(완전히 벗어난 탭) 처리**: 겹침이 전혀 없으면 겹친 너비가 0이 되고, 이는 스펙의
   "10px 미만이면 게임 오버" 규칙에 자동으로 포함된다. 별도의 "완전 미스" 분기를 만들지
   않고 하나의 규칙으로 통일했다.
3. **카메라(화면 스크롤)**: 타워가 화면 세로 길이를 넘어서면, 최근 6개 층만 보이도록 카메라가
   부드럽게(지수 감쇠 보간) 아래로 스크롤한다. 완전히 새 개념이 아니라 이 장르의 표준 UX이며,
   스크롤이 없으면 몇 층만 쌓아도 화면 밖으로 나가 게임을 계속할 수 없다.
4. **첫 이동 블록의 너비**: 기준 블록(0층)과 동일한 너비로 시작한다. 완전히 다른 너비로
   시작하면 첫 탭부터 극단적으로 잘릴 수 있어 스택 게임 특유의 "처음엔 쉽고 점점 어려워지는"
   곡선이 깨진다.
5. **잘려나간 조각의 시각 효과**: 겹치지 않은 나머지 부분은 살짝 아래로 떨어지며 사라지는
   애니메이션을 추가했다(스펙엔 없지만, "삐진 부분 잘림"을 시각적으로 보여주지 않으면 무슨
   일이 일어났는지 플레이어가 알기 어렵다).
6. **점수**: 스펙대로 "층 수"를 그대로 점수로 사용한다(기준 블록은 0층, 첫 성공 배치가
   1층). 별도의 정확도 보너스 등은 추가하지 않았다.

## 테스트 작성 중 발견한 타이밍 함정

"정확히 정렬됐을 때 탭하면 성공한다"를 검증하려고 처음엔 Node 쪽에서 `setTimeout` 폴링으로
정렬 여부를 확인한 뒤 별도의 `locator.click()` CDP 호출로 탭했다. 그런데 "정렬 확인"과
"실제 탭"이 서로 다른 왕복(round trip)이라, 그 사이에도 실시간 애니메이션은 계속 진행돼
탭이 실제로 도착하는 시점엔 이미 큰 폭(관측상 최대 130px 이상)으로 어긋나 있는 레이스
컨디션이 있었다(할리갈리의 "서버 수신 순서" 문제와 본질적으로 같은 종류의 함정 —
`docs/party-games-handoff-2026-07-03.md` 참고). `page.waitForFunction`으로 바꿔도
"확인"과 "탭"이 여전히 별개의 CDP 호출이라 근본적으로 같은 문제가 남았다. 최종적으로
`page.evaluate()` 안에서 `requestAnimationFrame` 폴링으로 정렬을 확인하는 즉시 같은 틱에서
`canvas.dispatchEvent(new PointerEvent('pointerdown', ...))`까지 실행해, "확인"과 "탭"을
브라우저 안 하나의 프레임으로 묶고 나서야 안정적으로 통과했다. **교훈: 실시간 애니메이션을
정확한 타이밍에 조작하는 E2E 테스트는 "상태 확인"과 "액션 실행"을 반드시 같은
`page.evaluate()` 호출 안에서 처리해야 하고, Node 쪽에서 두 단계로 나누면 그 사이의 CDP
왕복 시간만큼 반드시 어긋난다.**
