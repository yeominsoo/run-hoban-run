# 색 맞추기 슬라이더 구현 노트 (2026-07-13)

## 원본 스펙 (v1)

- R/G/B 슬라이더를 드래그해 목표 색과 최대한 가깝게 맞추기
- 10라운드, 라운드당 15초
- 점수: 정확도 × 100 (라운드당 최대 100점)
- Canvas 사용(부분: 색상 미리보기 rect만), 슬라이더 터치 영역 44px 이상

## 결정과 이유

1. **슬라이더 구현체**: 커스텀 캔버스 슬라이더 대신 네이티브 `<input type="range">` 3개
   사용. 이유: 네이티브 range input은 터치/마우스/키보드(방향키)를 전부 공짜로 지원하고
   스크린리더 접근성도 기본 제공한다. "터치 영역 44px 이상" 요구사항도 CSS 패딩으로 쉽게
   맞출 수 있어, 굳이 캔버스로 직접 드래그 로직을 재구현할 이유가 없다. 트랙/썸 스타일만
   `-webkit-slider-thumb`/`-moz-range-thumb`로 테마에 맞게 커스터마이즈했다.
2. **정확도 공식**: RGB 3채널 유클리드 거리 기반 —
   `accuracy = max(0, 1 - dist(현재,목표) / maxDist)`, `maxDist = sqrt(255²×3) ≈ 441.67`.
   맨해튼 거리(채널별 절대오차 합)보다 유클리드 거리가 "색이 실제로 얼마나 달라 보이는지"에
   더 가깝다고 판단했다.
3. **라운드 제출 방식**: v1 스펙에 명시가 없어 새로 설계 — "확정" 버튼으로 조기 제출 가능,
   15초가 다 되면 그 시점 슬라이더 값으로 자동 제출. 이유: 정확히 맞췄다고 확신하는데
   15초를 억지로 채우게 하면 지루하다. 확정 시 즉시 다음 라운드로 넘어간다(총 플레이 시간에
   대한 별도 보너스는 없음 — v1 스펙이 정확도만 언급하므로 스피드 보너스를 임의로 추가하지
   않았다).
4. **라운드 시작 슬라이더 위치**: 매 라운드 128/128/128(중간 회색)로 초기화. 이유: 만약
   무작위 시작값을 쓰면 우연히 목표색과 가까운 곳에서 시작하는 라운드가 생겨 공정성이
   떨어진다. 고정된 중립값에서 시작해야 라운드 간 난이도가 균일하다.
5. **캔버스 사용 범위**: v1이 "Canvas 사용: 부분(색상 미리보기 rect만)"이라 명시했으므로,
   목표색·현재색 스와이프 미리보기만 `<canvas>`에 그리고 슬라이더 자체는 DOM으로 뒀다.
   실시간으로 슬라이더를 움직일 때마다 정확도%도 같이 갱신해 즉각적인 피드백을 준다(스펙엔
   없지만, 라운드가 끝나야만 결과를 알 수 있으면 게임의 재미가 크게 떨어지므로 추가).
6. **목표색 생성**: 라운드마다 R/G/B 각각 0~255 균등 난수. 너무 극단적인 순백/순흑 등을
   피하는 보정은 넣지 않았다 — v1에 그런 제약이 없고, 극단색도 똑같이 유효한 학습 대상이다.
