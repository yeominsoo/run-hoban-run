## 프로젝트 개요

- **프로젝트명**: run-hoban-run(허브 타이틀 `Toris Arcade`) 게임 모음집 확장
- **목표**: 기존 9종 행사/파티 게임(경마 추첨·팀배분·주사위·가위바위보·라이어게임·마피아게임·
  할리갈리·윷놀이·전략윷놀이)에 이어, 짧게 즐기는 **미니게임 20종**(v1 유래 솔로 10종 +
  텐텐오락실 오마주 후보 10종)을 허브에 추가
- **기술 스택**: Vite + TypeScript + 순수 DOM / Canvas API (React 미도입, 외부 게임 라이브러리
  미도입 — 기존 규칙 그대로 유지). Three.js는 이미 프로젝트 의존성에 있지만 이번 미니게임들은
  전부 2D이므로 사용하지 않는다. 단, [QR/바코드 대기실 접속 개선](#qr바코드-대기실-접속-개선)을
  위해 경량 QR 인코딩 라이브러리(`qrcode`, MIT) 1종만 신규 추가한다 — 이는 게임 로직이 아닌
  공용 인프라 유틸이라 "외부 게임 라이브러리 금지" 규칙의 대상이 아니다.
- **배포**: 기존과 동일한 Firebase Hosting 파이프라인(`npm run deploy:firebase`), 신규 WAS/
  WebSocket 인프라 불필요
- **작성일**: 2026-07-13 (v1 대체)

## 현재 상태와 달라지는 점

v1은 "허브를 새로 만든다"는 전제였지만, 실제로는 이미 `src/pages/hub/main.ts`에 게임 카드
배열(`GAMES`)과 `hub.css` 아이콘 체계가 있고 9개 게임이 등록되어 있다. 따라서:

- **허브를 새로 만들 필요 없음.** `GAMES` 배열에 항목 10개를 추가하고 `hub.css`에 아이콘
  클래스(`.aim-trainer-mark` 등)만 추가하면 된다.
- **v1에서 가져온 신규 10종은 전부 로컬 단일 플레이**로 설계한다 (기존 `race`/`team`/`dice`와
  같은 계열). 실시간 대전이 필요 없으므로 `ws-server/`, `.env`의 `VITE_*_WS_URL`, 배포
  매니페스트의 WS env 추가가 전부 불필요하다.
- **[텐텐오락실 오마주 후보](#텐텐오락실-오마주-후보) 10종은 반대로 전부(합이 10 퍼즐의
  로컬 연습 모드 제외) 다인/1:1 실시간(WS) 게임으로 설계한다.** 텐텐오락실 자체가 "모일수록
  재밌는" 방 기반 동시 참여 앱이기 때문이다. 기존 `rps`/`liar`/`mafia`/`halligalli`/`yutnori`/
  `strategy-yutnori`처럼 WS 서버 확장이 필요하며,
  `docs/party-games-handoff-2026-07-03.md`의 "새 WS 게임 추가 절차"를 그대로 따른다.
- **점수 저장은 `localStorage`**로 기존 계획과 동일하게 유지한다 (`rhh_<game>_best` 키).
- **Capacitor / Google Play 출품**은 이 저장소에 전례가 없는 별도 사업 결정이라 확정 계획에서
  분리해 [보류 항목](#보류-항목--google-play-출품)으로 내렸다.

## 공통 시스템

### 재사용 가능한 기존 `src/shared/` 자원

- `seed.ts` — 시드 기반 RNG (`randomSeed`, `rollValues` 등). 신규 게임 중 시드 재현이 의미
  있는 것(예: 없음, 대부분 실시간 반응속도 게임)은 없지만 필요 시 그대로 가져다 쓸 수 있다.
- `participants.ts` — 참가자 목록 로드/저장. v1 유래 10종은 단일 플레이어라 대부분 불필요하지만,
  텐텐오락실 오마주 후보 10종은 `liar`/`mafia`처럼 방 참가자 목록·닉네임을 다루므로 재사용한다.
- `center-toast.ts/css` — 결과/오류 토스트. 게임오버 안내 등에 재사용.
- `pastel-theme.css` — 기존 9종 게임과 시각 톤을 통일하기 위해 신규 게임 CSS도 이 팔레트를
  기준으로 작성한다.

### 신규로 추가할 공용 자원

- `src/shared/pointer.ts` — touch/mouse 통합 좌표 및 이벤트 헬퍼. v1의 `shared/utils.js`
  아이디어를 TypeScript로 이식:
  ```ts
  export function getPointerPos(e: PointerEvent | TouchEvent): { x: number; y: number };
  export function onTap(el: HTMLElement, cb: () => void): void;
  export function onSwipe(el: HTMLElement, cb: (dir: 'up'|'down'|'left'|'right') => void): void;
  ```
- `src/shared/score-store.ts` — `localStorage` 최고점수 저장/조회 (`rhh_<game>_best` 키),
  방치형 농장의 오프라인 수익 계산에 필요한 "마지막 접속 시각" 저장도 이 모듈에 포함.

### 게임 쉘 UI

별도 공통 컴포넌트를 새로 만들기보다 기존 패턴을 게임 성격별로 그대로 따른다.

- **v1 유래 10종(로컬 단일 플레이)**: `dice`/`team`이 쓰는 `sidebar` + `back-link`
  (`← 게임 선택`) 레이아웃 대신, 사이드바 없이 캔버스 중심으로 상단에 `← 게임 선택` 링크 +
  게임명 + 최고점수만 있는 얇은 헤더바를 게임마다 반복 작성한다(공통 컴포넌트화는 3개 이상
  게임에서 중복이 확인된 뒤 판단).
- **텐텐오락실 오마주 후보 10종(WS 다인/1:1)**: `liar`/`mafia`의 로비 화면 패턴(닉네임
  입력 → 방 만들기/참가하기 → 참가자 리스트 + 호스트 시작 버튼 → `room-share` 영역)을 그대로
  재사용하고, [QR/바코드 대기실 접속 개선](#qr바코드-대기실-접속-개선)의 QR 진입 흐름을 처음부터
  포함한다.

## 게임별 명세

룰과 판정 기준은 v1과 동일하다 (요약만 유지, 상세 문구는 `docs/game-collection-plan-2026-07-13.md`
참고). Canvas 구현은 각 게임의 `src/pages/<name>/main.ts`에서 `<canvas>` 엘리먼트를 직접
다룬다 (외부 렌더링 라이브러리 없음).

| 게임 | 입력 | 핵심 규칙 | Canvas | 모바일 주의사항 |
|---|---|---|---|---|
| 에임 트레이너 | 탭/클릭 | 30초 제한, 원 크기 80→30px로 축소 | 예 | 없음 |
| 색 맞추기 슬라이더 | 슬라이더 드래그 | 10라운드×15초, RGB 오차→정확도% | 부분 | 슬라이더 터치 영역 44px 이상 |
| 볼 피하기+수집 | 드래그 | HP3, 빨간공 -1 / 초록공 +10, 30초마다 난이도↑ | 예 | 없음 |
| 타워 쌓기 | 탭 1회 | 겹치는 부분만 유지, 너비 10px 미만 시 종료 | 예 | 없음 |
| 스네이크 비틀기 | 방향키/스와이프 | 벽 없음(반대편 통과), 속도업마다 색 테마 변경 | 예 | 스와이프 방향 감지 |
| 타이핑 생존 | 키보드 | 낙하 단어 타이핑, HP3, 10초마다 난이도↑ | Canvas+DOM input | `visualViewport` 기준 높이 재계산, 모바일은 영어 전용 권장 |

## 텐텐오락실 오마주 후보

Google Play/App Store에 출시된 파티게임 앱 "텐텐오락실"(개발사: AppteenPlanet Co., Ltd.)을
조사한 결과를 반영한 신규 후보군이다. 게임 규칙·장르는 저작권법상 보호 대상이 아닌 "아이디어"
영역이고(구체적 표현만 보호), 아래 후보는 대부분 텐텐오락실도 원조가 아닌 보편적 파티게임/
캐주얼게임 장르(두더지잡기, 가위바위보류 반응게임, 업다운 추론, 숫자 퍼즐 등)라 오마주 개발
자체는 통상적인 관행이다. 다만 명칭·아트·카피는 반드시 독자 제작한다 —
자세한 기준은 [오마주 안전장치](#오마주-안전장치) 참고.

**전부 다인 실시간(WS) 게임으로 설계한다.** 텐텐오락실 자체 카테고리를 보면 "혼자하기"로
분류된 건 사과게임 하나뿐이고, 나머지(두더지·순서대로·기억력게임·산수홀짝·구구단·비번맞히기
등)는 전부 "빠르게"/"생각해" 카테고리 — 방에 모인 여러 명이 동시에 참여해 반응속도·정확도로
실시간 랭킹을 겨루는 구조다. "모일수록 재밌는" 것이 텐텐오락실의 핵심 정체성이므로, 이 저장소도
로컬 단일 플레이가 아니라 이 실시간 대결 구조 그대로 오마주하고, [QR/바코드 대기실 접속
개선](#qr바코드-대기실-접속-개선)으로 설계한 "QR로 방에 모여 곧장 입장" 흐름과 맞물리게 한다.

## 오마주 안전장치

위 후보를 실제로 개발할 때 지켜야 할 원칙. 일반적인 저작권법 상식 수준의 정리이며 법률
자문을 대체하지 않는다 — Google Play 정식 출시를 확정하는 시점에는 별도로 전문가 검토를
받는다([보류 항목](#보류-항목--google-play-출품)와 동일한 원칙).

- **명칭·상표**: "텐텐오락실", "TenTen" 등 원본 브랜드명·로고를 게임명이나 UI에 절대 쓰지
  않는다. 위 표의 "가칭"도 실제 확정 전 키프리스 등에서 상표 검색을 권장한다(본 저장소
  `AGENTS.md`의 "우마무스메 명칭·UI 복제 금지" 원칙과 동일선상).
- **협찬 버전 배제**: "카스" 등 제3자 브랜드와 협업한 버전(탭탭cass 등)은 참고 대상에서
  완전히 제외한다(상표권 + 주류 광고 규제 이중 리스크).
- **표현물은 전부 독자 제작**: UI·그래픽·사운드·카피 문구는 텐텐 앱 스크린샷/에셋을 참고하지
  않고, 본 저장소의 `pastel-theme.css` 톤에 맞춰 새로 만든다. 슬로건 문구("뭐가 나올지
  아무도 몰라몰라몰라" 등)도 그대로 베끼지 않는다.
- **룰 수치도 자체 정의**: 제한시간·점수 배율·난이도 곡선 등 세부 수치는 텐텐과 동일하게
  맞추지 않고 이 저장소의 밸런스로 새로 정한다(예: 두더지 사냥 제한시간을 텐텐과 다르게 설정).
- **코드는 처음부터 독자 구현**: 텐텐 소스코드에 접근한 적이 없으므로 자연히 충족되지만,
  구현 담당자도 텐텐 앱의 실행 파일을 디컴파일하거나 참고하지 않는다 — 이 문서의 "룰 요약"
  수준의 설명만 참고해서 새로 설계한다.

## QR/바코드 대기실 접속 개선

### 현재 상태

기존 실시간 게임(`rps`/`liar`/`mafia`/`halligalli`/`yutnori`/`strategy-yutnori`)은 6자리
방 코드 + `src/shared/share.ts`의 `shareRoomLink()`(Web Share API 우선, 미지원 시 클립보드
복사)로 초대 링크를 공유한다. 참가자는 `?room=<CODE>` 파라미터로 링크를 열면 방 코드 입력
칸이 자동으로 채워지지만, 그 다음에도 참가 버튼 클릭과 닉네임 입력을 별도로 거쳐야 한다.

### 목표

행사 현장처럼 **근거리에 모인 참가자**가 호스트 화면의 QR 코드 하나를 스캔해서 방 코드 입력
단계 없이 곧장 닉네임 입력까지 이어지도록 마찰을 줄인다. 원격 참가자를 위한 링크 공유 방식은
그대로 유지한다.

### 방식 결정: QR 코드로 통일 (1D 바코드는 채택하지 않음)

1D 바코드는 6자리 방 코드 문자열만 인코딩할 수 있어 스캔 후에도 여전히 앱을 직접 열고
코드를 입력하는 절차가 남는다. 반면 QR 코드는 `https://<host>/<game>/?room=<CODE>` 전체
URL을 인코딩할 수 있어 스캔 즉시 해당 게임의 참가 화면으로 바로 이동한다. 또한 대부분의
스마트폰 기본 카메라 앱이 QR은 자동 인식하지만 1D 바코드는 별도 스캐너 앱이 필요한 경우가
많다. 두 가지 이유로 **QR 코드 단일 방식**으로 구현하고 1D 바코드는 채택하지 않는다.

### 구현안

1. **`src/shared/qr-code.ts` 신규** — 경량 QR 인코더 라이브러리(`qrcode` npm 패키지, MIT
   라이선스, 완전 오프라인 canvas/SVG 렌더링, 네트워크 호출 없음)를 `package.json`
   dependencies에 추가하고, 그 위에 얇은 래퍼(`renderRoomQr(canvas, url)`)를 만든다. 신규
   미니게임에 적용한 "외부 라이브러리 금지" 규칙은 게임 로직 코드에 한정되며, 이런 공용
   인프라 유틸(이미 `three`가 있듯)에는 적용하지 않는다.
2. **로비 화면 확장** — 각 게임의 `room-share` 영역(예: `liar/main.ts`의
   `#lobby-share`)에 기존 "링크 복사" 버튼 옆에 "QR로 초대" 탭/토글을 추가해 QR 이미지를
   캔버스로 렌더링한다.
3. **QR 스캔 진입 흐름 단축** — `?room=<CODE>`로 진입했을 때 현재처럼 입력 필드만 채우는
   대신, 방 코드 입력 UI 자체를 숨기고 곧바로 "닉네임을 입력하고 입장하기" 화면으로 전환한다.
4. **닉네임 마찰 감소** — 마지막으로 사용한 닉네임을 `localStorage`에 저장해두고(신규
   `src/shared/nickname.ts` 또는 기존 `participants.ts` 확장) 입력 필드 기본값으로
   채워, 재방문 시 원터치에 가깝게 입장할 수 있게 한다.
5. **세 가지 입장 경로 유지** — QR(근거리 최적화) / 링크 공유(원격 참가자) / 6자리 코드
   수동 입력(스캔 불가 상황의 폴백)을 모두 남겨 접근성을 해치지 않는다.
6. **적용 범위** — 기존 실시간 게임 6종(`rps`/`liar`/`mafia`/`halligalli`/`yutnori`/
   `strategy-yutnori`)과 텐텐오락실 오마주 후보 10종(합이 10 퍼즐의 WS 방 모드 포함) 로비
   화면에 공통 적용한다. 기존 게임을 건드리는 변경이므로 `npm run test:render`
   (Playwright)로 회귀 여부를 반드시 확인한다.

## 모바일 대응

기존 게임들(윷놀이 등)이 이미 모바일 대응을 하고 있는 패턴을 그대로 따른다.

- 루트 `index.html`의 viewport 메타(`width=device-width, initial-scale=1.0`)는 이미
  적용되어 있으므로 신규 게임 `index.html`도 동일하게 작성
- Canvas 반응형 크기 계산은 게임 헤더바 높이를 제외한 `window.innerWidth/innerHeight` 기준
- 터치 이벤트는 `touchstart`/`touchmove`/`touchend` 모두 `preventDefault()`로 스크롤 방지,
  `shared/pointer.ts`의 `getPointerPos()`로 좌표 통일, 탭 영역 최소 44×44px

## 세션 간 협업 규칙

`AGENTS.md`의 기존 규칙을 그대로 따른다. 이번 계획 관련해 특히 유의할 점:

- **GitHub SSH**: WSL2에서 fetch/push 시 `GIT_SSH_COMMAND='ssh -i
  /home/msyeo/.ssh/yeominsoo_ed25519 -o IdentitiesOnly=yes'`를 명시한다.
- **병렬 세션 확인**: 이 저장소는 여러 세션이 동시에 작업한 이력이 있다. 특히 `vite.config.ts`,
  `firebase.json`, `src/pages/hub/main.ts`는 게임을 하나 추가할 때마다 공통으로 건드리는
  파일이라 충돌 가능성이 높다 — 작업 전 `git fetch && git log origin/master --oneline -10`으로
  확인하는 습관을 유지한다.
- **브랜치 전략**: `master`는 완성된 기능만 병합, 게임별로 `feature/{game-name}` 독립 브랜치
  사용, PR 병합 전 기준은 위 [개발 단계](#개발-단계)의 완료 기준 체크리스트.
- **코딩 규칙**: 게임 로직에는 외부 라이브러리 사용 금지(Vanilla TS + Canvas API만, `three`
  미사용) — 단 `qrcode`처럼 게임 로직이 아닌 공용 인프라 유틸은 예외([QR/바코드 대기실
  접속 개선](#qr바코드-대기실-접속-개선) 참고). 주석은 WHY가 비자명할 때만 최소한으로(레포
  공통 스타일), `src/shared/`의 재사용 유틸을 적극 활용해 중복 코드 방지.
- **게임 상태 업데이트**: 게임 하나 완료할 때마다 이 파일의
  [포함 게임](#포함-게임-v1-유래-신규-10종--텐텐오락실-오마주-후보-10종-기존-9종은-이미-완성)
  표 상태를 `⬜ 미개발 → 🚧 개발 중 → ✅ 완료`로 갱신한다.
