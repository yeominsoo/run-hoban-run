# 리버시(reversi) 구현 노트 (2026-07-16)

텐텐오락실 오마주 10종 완료 직후 사용자가 "오델로도 있으면 좋겠다"고 요청해 추가한
게임. 계획서의 텐텐오락실 오마주 계열과 무관한 독립 요청이라 별도 Phase 구분 없이
바로 진행했다.

## 게임명을 "오델로"가 아니라 "리버시"로 정한 이유

사용자는 "오델로"라고 불렀지만, **"Othello"는 실제 상표**(현재 Mattel/Pressman 계열이
보유)이고 게임 규칙 자체의 공용 명칭은 "Reversi"다 — 체스/장기처럼 규칙은 누구나 구현할
수 있지만 "오델로"라는 이름은 브랜드다. 이 저장소가 이미 지켜온
"우마무스메 명칭·UI 미사용"(`AGENTS.md`), "텐텐오락실 브랜드명 미사용"(오마주
안전장치) 원칙과 같은 선상에서, 게임명·UI 문구에는 "리버시"만 쓰고 사용자에게도
간단히 안내했다. 규칙 자체(8x8 보드, 감싸서 뒤집기)는 특정 회사의 지적재산이 아니라
공개된 보드게임 규칙이라 그대로 구현하는 데 문제가 없다.

## 플레이 방식 — 온라인 1:1 실시간(사용자 선택)

로컬 pass-and-play(한 기기에서 번갈아 두기)와 온라인 1:1 실시간 중 사용자에게 직접
물어 온라인 방식을 선택받았다. 줄다리기 배틀·영역 쟁탈전에서 확립한 "로비 대기 없이
join 즉시 자동 매칭 + 3초 카운트다운" 골격을 그대로 재사용했다.

## 스펙에 없어 직접 정한 수치

- **자기 차례 타임아웃 30초**: 리버시는 실시간 반응 게임이 아니라 생각할 시간이
  필요한 턴제 전략 게임이라, 줄다리기 배틀(30초 전체 매치)이나 라이트 게스(1.2초
  빨간불)보다 넉넉하게 잡았다. 전략윷놀이의 "자기 차례 20초 타임아웃" 정책을 참고해
  좀 더 여유 있게 30초로 정했다.
- **타임아웃 시 강제 종료가 아니라 무작위 합법수 자동 착수**: 게임을 중단시키지 않고
  계속 진행되도록 하는 편이 상대방 경험에 더 낫다고 판단했다 — 전략윷놀이의 타임아웃
  자동 처리 정책과 동일한 철학.

## 서버 권위 원칙

리버시는 비밀 정보가 전혀 없는 완전 정보 게임(양쪽 다 보드 전체를 항상 볼 수 있음)이라
"정보를 숨기는" 안티치트는 필요 없다. 대신 "합법수 판정"이 유일한 신뢰 포인트다 —
클라이언트가 `place {row,col}`만 보내면 서버가 8방향 전부를 탐색해 실제로 뒤집히는
돌을 계산하고, 그 계산 결과(확정 보드 전체)만 브로드캐스트한다. 클라이언트도 UI
힌트(둘 수 있는 칸 강조)를 위해 같은 로직을 자체 구현해 실행하지만, 이건 순전히
표시용이고 실제 착수 처리는 항상 서버 재계산을 통과해야 한다 — 클라이언트의 힌트
로직과 서버의 판정 로직이 어긋나도(예: 버그) 서버가 최종 권위이므로 게임 결과가
깨지지 않는다.

## 검증

- **raw WebSocket 스크립트**: 초기 보드가 표준 리버시 배치(중앙 4칸 흑백 대각선)로
  정확히 생성되는지 확인. 흑이 표준 오프닝 수(2,3)를 두면 정확히 1칸(3,3)이 뒤집히고
  점수가 4:1이 되는지 확인. 백이 같은 자리에 올바르게 이어서 둘 때도 뒤집기가 정확한지
  확인. 규칙상 뒤집을 대상이 없는 좌표(0,0)에 착수를 시도하면 **어떤 브로드캐스트도
  발생하지 않고 조용히 무시**되는지 확인(첫 시도에서 소켓 큐에 남아있던 이전 라운드의
  브로드캐스트를 잘못 읽어 "불법 착수가 통과된 것처럼" 보였는데, 큐를 완전히 비우고
  재검증한 결과 서버가 처음부터 정확히 거부하고 있었음을 확인 — 영역 쟁탈전 검증 때와
  동일한 종류의 테스트 스크립트 함정이었다).
- **Playwright UI 테스트**: 모바일 세로(390×844)에서 호스트(흑)가 방을 만들고
  게스트(백)가 참가하면 자동 매칭 → 카운트다운 → 보드 화면까지 진행해, 표준 오프닝
  수를 탭으로 두면 실제 화면에서 돌이 뒤집히고 점수(4:1)와 차례 안내 문구가 양쪽
  클라이언트에 정확히 동기화되는지 확인했다.

## 배포

WAS 배포는 앞선 게임들과 동일한 절차(`ws-server/README.md`)를 따른다.
