# 그래픽 리워크 기획 및 개발 계획서 - 2026-06-12

## 1. 개요

현재 `run-hoban-run`은 기능적으로 레이스와 토너먼트가 동작하지만, 화면은 경마 게임보다 디버그 시뮬레이터처럼 보인다. 문제는 단순히 에셋 품질이 낮은 것이 아니라 카메라, 트랙, 모델, 라벨, 검증 기준이 경마 문법을 만들지 못하는 데 있다. 리워크의 목표는 리얼리즘이 아니라 "스타일라이즈드 방송형 경마"로 방향을 고정하고, 다음 구현 세션이 카메라와 화면 문법부터 안정적으로 개선하도록 만드는 것이다.

## 2. 현재 문제 진단

### 화면 구도

- 레이스 시작 후 트랙이 화면 대부분을 사선으로 덮고, 말과 기수는 작은 반복 오브젝트처럼 보인다.
- 출발 전 화면은 3D 장면보다 빈 흰 영역과 양쪽 패널이 먼저 보인다.
- 화면의 중심이 "말들이 달리는 장면"이 아니라 "넓은 주황색 주로와 라벨 묶음"에 있다.

### 카메라

- 현재 기본 카메라는 `overview` 중심이며, 높은 위치에서 전체 주로를 내려다본다.
- `getOverviewCameraView`, `getFocusedCameraView`, 헬리콥터/스킬 컷신 카메라가 모두 `src/main.ts` 안에 섞여 있다.
- 낮은 3/4 팩 추적, 선두 후방 추적, 결승선 고정 컷 같은 경마 중계 카메라가 기본 흐름으로 설계되어 있지 않다.

### 트랙

- `trackVisualLength`, `trackVisualWidth` 기반의 긴 직선 주로가 육상 트랙처럼 읽힌다.
- 모든 참가자 수에 맞춰 흰 레인 라인이 늘어나면서 경마장보다 달리기 레인처럼 보인다.
- 출발 게이트, 거리 표지, 안쪽/바깥쪽 레일, 결승 포스트, 관중석 같은 경마장 맥락이 약하다.

### 모델

- 현재 기본 전략은 `poly-google-bridle-horse.glb`와 `eclair-sitting-rider.glb`를 조합하고, 실패 시 절차형 fallback을 사용한다.
- 말과 기수의 스케일, 포즈, 애니메이션 톤이 하나의 경주용 캐릭터 세트처럼 맞지 않는다.
- 그래픽 방향이 리얼 GLB와 절차형 캐릭터 사이에 걸쳐 있어 의도된 스타일로 보이지 않는다.

### UI

- `.runner-tag`가 모든 말에 붙어 있고, `positionRunnerLabels`가 화면 안으로 라벨을 밀어 넣으면서 장면을 덮는다.
- 이름은 leaderboard와 말 옆구리 nameplate에서 확인하므로, 말 위의 큰 상시 라벨은 정보 중복이자 시각 방해다.
- 레이스 중 화면에서 중요한 정보의 우선순위가 없다. leader, selected runner, skill event, winner가 같은 라벨 체계 안에 섞인다.

### 테스트 기준

- `tests/render.spec.ts`의 첫 검증은 nonblank canvas와 색상 변화량 중심이다.
- 이후 테스트는 에셋 상태, 헬리콥터 진단용 `data-*`, 라벨 존재, 특정 컷신 상태에 많이 결합되어 있다.
- 현재 테스트는 "화면이 경마처럼 보이는가"를 확인하지 못한다.
- 테스트를 위해 제품 DOM/UI/runtime을 바꾸는 방식은 금지한다. 검증은 실제 사용자 화면을 기준으로 설계해야 한다.

## 3. 목표 비주얼 방향

"스타일라이즈드 방송형 경마"는 리얼 경마 시뮬레이터가 아니라, 대형 현장 디스플레이에서 한눈에 경마처럼 읽히는 캐주얼 3D 레이스 연출이다. 말과 기수는 통일된 저폴리 또는 마스코트 스타일로 크고 선명하게 보이며, 화면은 경마장, 레일, 출발 게이트, 결승 포스트, 관중석, 먼지, 접지 그림자, 중계형 카메라 컷으로 구성된다.

좋은 첫 화면은 빈 캔버스가 아니라 출발 게이트 또는 라인업 장면을 보여준다. 좋은 경주 중간 화면은 선두와 주변 팩이 크게 잡히고, desktop에서는 우측 세로 leaderboard가 현재 순위를 담당하며, mobile에서는 하단 leaderboard가 최소 정보만 보조한다. 캔버스에는 선택 주자나 스킬 발동 같은 중요한 콜아웃만 뜬다. 좋은 결승 장면은 결승 포스트와 선두 말이 화면 중심에 있고, 포토피니시 또는 위너서클 연출로 결과를 자연스럽게 받아들일 수 있어야 한다.

## 6. UX 및 화면 구성 계획

대형 현장 디스플레이 기준으로 레이스 캔버스가 첫 번째 정보가 되어야 한다. 설정 패널은 출발 전에는 운영 도구로 보이되, 경기 시작 후에는 자동으로 숨겨지고 필요할 때만 열어야 한다. 레이스 중에는 화면 대부분이 말과 경마장으로 채워져야 하며, desktop 우측 leaderboard, mobile 하단 leaderboard, 제한된 콜아웃이 정보를 보조한다.

권장 역할 분담:

- 설정 패널: 참가자, 순서변경 버튼, 출전/진출/우승 수 입력만 담당한다.
- 레이스 캔버스: 말, 경마장, 카메라 컷, 스킬/결승 연출을 담당한다.
- leaderboard: desktop에서는 경기 정보 아래 우측 세로 스크롤 목록, mobile에서는 하단 가로 목록으로 참가자 이름, 현재 순위, 선택 대상, 탈락/진출/우승 상태를 제공한다.
- on-canvas callout: leader, selected runner, skill event, winner만 표시한다.
- 결과 정보: 별도 결과 목록을 두지 않고, 현재 순위 목록이 레이스 종료 후 시간/진출/탈락 상세를 함께 제공한다.
- minimap: 전체 위치 파악용으로 유지하되, 경주 장면보다 강하게 보이면 안 된다.

말 이름과 순위 표시 기준:

- 모든 말 위 이름표 상시 노출은 제거한다.
- 기본 상태에서는 색상, 무늬, 말 옆구리 nameplate 이름으로 말을 구분한다.
- 이름과 순위의 전체 목록은 leaderboard에서 확인한다.
- 캔버스 콜아웃은 동시에 4개 이하를 목표로 한다.
- 스킬 발동 시에는 해당 말 위에 짧은 콜아웃을 띄우고, 종료 후 사라지게 한다.
- 결승 직후에는 1위 또는 우승자만 강조한다.

## 7. 3D 장면 리워크 계획

### 트랙/경마장

- 현재 긴 직선 박스 트랙을 "경마장 직선 구간"으로 다시 읽히게 만든다.
- 레인 라인은 출발 정렬용 보조선 수준으로 줄이고, 레이스 중에는 경마장 레일과 주로 질감이 우선 보이게 한다.
- 주로 색상은 단일 모래 주로로 고정한다.
- 안쪽/바깥쪽 레일, 결승 포스트, 거리 표지, 출발 게이트, 관중석 실루엣을 추가한다.
- 말이 밟는 위치에는 얕은 먼지, 발굽 자국, 접지 그림자를 넣는다.

### 말/기수 모델

- 권장 기본값은 절차형 stylized horse/rider다.
- 기존 GLB는 즉시 삭제하지 않고, Phase 2에서 모델 전략을 확정한 뒤 fallback 또는 실험용으로 격하한다.
- 말은 큰 몸통, 명확한 목/머리/다리, 과장된 갈기/꼬리, 말 옆구리 nameplate를 가진다.
- 기수는 헬멧, 상의 색상, 팔/다리 실루엣이 보이는 단순 모델로 통일한다.
- 참가자별 개성은 색상, 무늬, 이름 nameplate로 표현한다.

### 애니메이션

- 첫 목표는 자연스러운 관절 정확도가 아니라 "달리는 느낌"이다.
- 말 몸통 bob, 앞/뒤다리 위상차, 꼬리 흔들림, 먼지 타이밍을 맞춘다.
- 속도 구간 배율은 이동 속도뿐 아니라 다리 모션 속도와 먼지 빈도에도 반영한다.
- 스킬 포즈는 경마 장면을 망가뜨리지 않도록 짧고 크게 보이되, 기본 주행 실루엣을 우선한다.

### 카메라 컷

- 카메라는 모드가 아니라 시퀀스로 설계한다.
- 기본 시퀀스:
  1. pre-race: 출발 게이트/라인업 정면 또는 3/4 뷰.
  2. early: 측면 중계 카메라로 팩 전체와 선두권을 보여준다.
  3. mid-race: 낮은 3/4 팩 추적 카메라로 선두와 추격 그룹을 크게 잡는다.
  4. final-stretch: 선두 또는 선택 주자 후방/측후방 추적 카메라로 속도감을 만든다.
  5. finish: 결승 포스트 고정 또는 포토피니시 카메라.
  6. winner: 우승자 클로즈업 또는 위너서클.
- 사용자가 leaderboard에서 주자를 선택하면 일반 주행 중에는 해당 주자를 중심으로 잡는다.
- 헬리콥터/스킬 컷신은 기본 경주 컷 시퀀스를 끊지 않고 짧게 삽입한다.

### 이펙트

- 1차 이펙트는 먼지, 접지 그림자, 속도감 라인, 결승 플래시다.
- 스킬 이펙트는 말/기수를 가리지 않는 위치와 지속 시간으로 제한한다.
- 헬리콥터 효과는 재미 요소로 유지 가능하지만, 경마장 품질이 안정된 뒤 조정한다.

### 조명/그림자

- 현재 hemisphere + directional light 구조는 유지하되, 접지 그림자와 말 하단 contrast를 강화한다.
- 배경 하늘과 주로 색상 대비를 낮춰 말과 기수가 먼저 보이게 한다.
- 대형 화면에서 과도한 흰색 glow와 빈 배경이 보이지 않도록 출발 전 장면을 구성한다.

### 모바일/데스크톱 대응

- 데스크톱은 1440x900 대형 화면 기준으로 먼저 잡는다.
- 모바일은 정보량을 줄이고, leaderboard와 minimap이 캔버스 주요 피사체를 가리지 않게 한다.
- 모바일에서 카메라가 과하게 멀어지지 않도록 별도 final-stretch/pack framing 값을 둔다.

## 8. 모델 전략

### A안: 절차형 stylized horse/rider 직접 구성

장점:

- 현재 Three.js 구조와 잘 맞고 외부 라이선스 리스크가 작다.
- 말/기수/스킬 포즈/색상/nameplate를 같은 스타일로 통제할 수 있다.
- 성능과 로딩 시간을 예측하기 쉽다.
- "웃긴 행사 게임" 톤에 맞춰 과장된 모델을 만들 수 있다.

단점:

- 초기 모델링 품질을 코드로 끌어올리는 시간이 필요하다.
- 리얼한 말 움직임을 기대하면 한계가 있다.

권장 판단: 기본 전략으로 채택한다. 이 프로젝트는 리얼리즘보다 일관된 화면 문법과 행사장 가독성이 중요하다.

### B안: 라이선스가 맞는 통일 asset pack 사용

장점:

- 제대로 된 horse/rider/rig/animation pack을 찾으면 빠르게 품질을 높일 수 있다.
- 갤럽 애니메이션 품질은 절차형보다 나을 수 있다.

단점:

- 무료 에셋을 개별 조합하면 현재와 같은 어색함이 반복된다.
- 라이선스, 파일 크기, 로딩 시간, 모바일 성능, 스킬 포즈 확장성 확인이 필요하다.

허용 조건:

- horse, rider, rig, animation이 같은 스타일 또는 같은 pack이어야 한다.
- 상업/행사 사용 가능 라이선스가 명확해야 한다.
- local asset으로 포함 가능해야 하며 runtime 외부 요청을 만들지 않아야 한다.
- Playwright와 수동 QA에서 20마리 동시 렌더링 성능이 확인되어야 한다.

완료 기준:

- 다음 구현자가 "무엇이 좋아져야 하는지"를 스크린샷 단위로 비교할 수 있다.

### Phase 1: 카메라/트랙/라벨 제거 중심의 빠른 시각 개선

목표: 모델 교체 전에도 디버그 시뮬레이터 느낌을 줄인다.

작업 항목:

- `overview` 기본 카메라를 높은 고각에서 낮은 3/4 pack view로 바꾼다.
- `focused` 카메라는 선택 주자가 화면 1/3 지점에 크게 잡히도록 조정한다.
- `updateRunnerLabel`/`positionRunnerLabels`를 바꿔 모든 runner tag 상시 노출을 중단한다.
- leader, selected, skill, eliminated/winner 상태만 콜아웃으로 표시한다.
- 흰 레인 라인은 제거하고 경마장 레일/거리 표지를 우선한다.
- 출발 전 빈 화면 대신 출발 게이트 또는 라인업 구도를 보이게 한다.

산출물:

- 카메라 framing 개선.
- 콜아웃 개수 제한.
- 경마장 문법이 보이는 기본 주로.

완료 기준:

- desktop mid-race 스크린샷에서 말/기수가 화면의 주요 피사체로 보인다.
- 캔버스 위 콜아웃이 동시에 4개 이하로 유지된다.
- leaderboard가 순위와 이름 정보의 기준 소스로 동작한다.
- 기존 토너먼트 흐름은 유지된다.

### Phase 2: stylized horse/rider 모델과 gallop 애니메이션

목표: 현재 GLB 조합의 어색함을 통일된 스타일로 대체한다.

작업 항목:

- 절차형 horse/rider 모델을 기본 시각 전략으로 확정한다.
- 이름 nameplate, 색상, 무늬를 말 모델에 직접 넣는다.
- 다리 위상차, 몸통 bob, 꼬리 흔들림, 접지 먼지를 gallop 루프로 묶는다.
- 기존 GLB asset 상태 테스트는 새 모델 전략에 맞게 계획적으로 수정한다.

산출물:

- 통일된 stylized horse/rider 모델.
- 속도 배율에 반응하는 gallop 애니메이션.
- GLB 의존성 축소 또는 명확한 fallback 정책.

완료 기준:

- 20마리 동시 렌더링에서 스타일이 일관된다.
- 말이 정지/주행/결승/탈락 상태에서 같은 모델 언어를 유지한다.
- 외부 runtime asset 요청이 없다.

### Phase 3: broadcast camera sequence와 final stretch/finish 연출

목표: 레이스가 자동 중계처럼 보이게 한다.

작업 항목:

- race progress를 기준으로 카메라 컷 시퀀스를 만든다.
- early/mid/final/finish 상태를 명시적으로 나누되, 제품 DOM을 테스트 전용 상태로 오염시키지 않는다.
- 결승선 근처에서 포토피니시 또는 finish post framing을 넣는다.
- 스킬/헬리콥터 컷신은 기본 시퀀스 위에 짧게 삽입한다.

산출물:

- broadcast camera sequence.
- final stretch speed framing.
- finish shot.

완료 기준:

- 같은 시드로 실행하면 주요 컷이 재현 가능하다.
- final-stretch 스크린샷에서 선두권 말과 결승 문법이 동시에 보인다.
- 헬리콥터 이벤트가 경주 흐름을 과도하게 가리지 않는다.

### Phase 4: 우승자 발표와 결과 캡처 품질 개선

목표: 결과 화면이 행사장에서 캡처/공유 가능한 장면이 되게 한다.

작업 항목:

- 우승자 banner와 canvas winner shot의 시각 우선순위를 정리한다.
- winner circle 또는 결승선 앞 우승자 클로즈업을 만든다.
- 결과 스크린샷/영상 캡처에서 라벨, leaderboard, winner banner가 서로 겹치지 않게 한다.

산출물:

- 우승자 발표 장면.
- 결과 스크린샷 품질 개선.

완료 기준:

- winner screenshot에서 우승자 이름, 말, 결승/우승 맥락이 함께 보인다.
- 모바일과 데스크톱에서 텍스트 겹침이 없다.

### Phase 5: visual regression 및 Playwright 품질 검증 강화

목표: nonblank canvas를 넘어 사용자-visible 품질을 검증한다.

작업 항목:

- 고정 seed 기반 pre-race, mid-race, final-stretch, finish/winner 스크린샷을 저장한다.
- 라벨/콜아웃 개수, UI 겹침, 주요 피사체 영역, desktop/mobile leaderboard 위치를 검증한다.
- 기존 헬리콥터/스킬 테스트는 제품 동작 검증으로 유지하되, 그래픽 품질 테스트와 분리한다.

산출물:

- visual regression screenshot set.
- 사용자-visible QA 체크리스트.

완료 기준:

- nonblank canvas만으로 pass하지 않는다.
- 화면 겹침, 콜아웃 과다, 주요 말 가시성 실패를 테스트 또는 수동 QA에서 잡을 수 있다.

## 10. 검증 계획

### Playwright 스크린샷 기준

- pre-race: 출발 게이트 또는 라인업이 보이고, 빈 흰 화면이 주요 영역을 차지하지 않는다.
- mid-race: 선두 또는 선택 주자와 주변 팩이 화면 중심에 크게 보인다.
- final-stretch: 결승 포스트/거리 표지와 선두권이 함께 보인다.
- finish/winner: 우승자 말, 이름, 결승/우승 맥락이 캡처된다.

### 자동 검증 후보

- canvas가 viewport 대부분을 차지한다.
- desktop leaderboard는 우측 세로 목록으로 스크롤되고, mobile leaderboard와 minimap은 서로 겹치지 않는다.
- visible `.runner-tag` 또는 새 callout 수가 4개 이하다.
- race 중 panel hidden 상태에서 주요 UI가 말/결승선 중심 영역을 가리지 않는다.
- desktop 1440x900, mobile 390x844 모두 스크린샷을 남긴다.

### 수동 QA 체크리스트

- 첫 화면을 보고 경마 게임임을 알 수 있는가?
- 시작 후 3초 안에 말과 기수가 화면의 주인공으로 보이는가?
- 모든 이름표가 말 위를 덮지 않는가?
- 선두와 추격 말의 관계를 눈으로 이해할 수 있는가?
- 결승 장면에서 누가 이겼는지 즉시 알 수 있는가?
- 웃긴 스킬/헬리콥터 연출이 경마 장면을 완전히 압도하지 않는가?
- 모바일에서 leaderboard, minimap, winner banner가 서로 겹치지 않는가?

## 11. 리스크 및 대응

| 리스크 | 대응 |
| --- | --- |
| 절차형 모델이 장난감처럼만 보일 수 있음 | 카메라, 먼지, 그림자, 이름 nameplate로 경마 문법을 먼저 보강한다. |
| GLB 제거/격하가 기존 테스트와 충돌 | Phase 2에서 테스트 기대값을 새 모델 전략에 맞춰 함께 갱신한다. 제품 동작을 테스트에 맞추지 않는다. |
| 카메라 컷이 멀미나 화면 혼란을 만들 수 있음 | 카메라 상태를 적게 시작하고, 컷 전환은 lerp와 짧은 hold를 사용한다. |
| 20마리 동시 렌더링 성능 저하 | 저폴리 모델, 공유 material, instance 후보를 검토한다. |
| 모바일에서 피사체가 너무 작아짐 | 모바일 전용 pack/final camera 값을 별도로 둔다. |
| 헬리콥터 이벤트가 경마 문법을 덮음 | 1차 리워크에서는 후순위로 두고, 기본 race framing이 안정된 뒤 조정한다. |
| 사용자가 기대하는 웃긴 연출과 경마 품질이 충돌 | 기본 레이스는 경마처럼, 스킬은 짧은 강조 컷으로 분리한다. |
| visual 테스트가 다시 data-* 중심으로 흐를 수 있음 | 스크린샷, 실제 DOM 배치, visible callout 수처럼 사용자-visible 기준을 우선한다. |

## 12. 우선순위 결론

바로 다음 구현 세션은 다음 5개 작업만 시작한다.

1. `P0`: 현재 스크린샷 baseline을 고정하고 visual target checklist를 만든다.
2. `P1A`: 기본 카메라를 낮은 3/4 pack view로 바꾸고 selected runner framing을 조정한다.
3. `P1B`: 모든 말 위 상시 이름표를 제거하고 leader/selected/skill/winner 콜아웃만 남긴다.
4. `P1C`: 주로의 흰 레인 라인을 줄이고 레일, 결승 포스트, 거리 표지, 출발 게이트를 추가한다.
5. `P1D`: Playwright에 pre-race/mid-race/final-stretch/winner 스크린샷 검증을 추가한다.

첫 구현 순서는 "카메라와 UI 간섭 제거 -> 경마장 문법 -> 모델 통일 -> 이펙트/연출 -> 테스트 강화"를 따른다.

단, 다음 구현 세션의 권장 완료 범위는 `P0 -> P1A -> P1B`다. `P1C`는 카메라와 콜아웃 정책이 안정된 경우에만 같은 세션에서 이어서 진행한다.
