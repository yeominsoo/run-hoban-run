# 에임 트레이너 구현 노트 (2026-07-13)

`docs/game-collection-plan-v2-2026-07-13.md`의 Phase 1 첫 번째 게임. 사용자가 "선택지가
필요할 때는 추천안으로 진행하고, 어떤 선택지에서 어떤 이유로 어떤 방식을 골랐는지 문서로
남겨달라"고 요청해서, 원본 계획서(v1)에 명시되지 않았거나 여러 구현 방식이 가능했던 지점을
정리한다.

## 원본 스펙 (v1, `docs/game-collection-plan-2026-07-13.md`)

- 목표: 화면에 원이 나타나면 빠르게 탭/클릭
- 판정: 원 안을 탭하면 명중 / 밖이면 미스
- 원 크기: 레벨 올라갈수록 작아짐 (80px → 30px)
- 시간: 30초 제한
- 점수: 명중 +100, 빠를수록 보너스
- Canvas 사용, 모바일 주의사항 없음

## 선택지와 결정

### 1. 터치/마우스 통합 방식

- **선택지**: (a) v1 계획서가 제안한 `touchstart`+`mousedown` 이중 바인딩, (b) Pointer Events
  단일 바인딩
- **결정**: (b) Pointer Events (`pointerdown`)
- **이유**: 저장소를 확인해보니 `src/main.ts`(레이스 리더보드 드래그)가 이미 Pointer Events를
  쓰고 있었다. 이 저장소가 타깃으로 하는 모든 최신 브라우저(모바일 Safari/Chrome, 데스크톱
  전반)가 Pointer Events를 지원하므로, 별도 브라우저 분기 없이 리스너 하나로 마우스·터치·펜을
  전부 처리할 수 있다. 기존 컨벤션과의 일관성이 더 중요하다고 판단해 v1의 제안 대신 이 방식을
  택했다.
- **반영**: `src/shared/pointer.ts`에 `getPointerPos()`/`onTap()`만 구현했다. v2 계획서가
  같이 적어둔 `onSwipe()`는 스와이프가 필요한 게임(스네이크·2048 변형)을 구현할 때 그 게임에서
  실제로 필요한 시그니처를 보고 추가한다 — 지금 당장 쓰이지 않는 함수를 미리 만들어두지 않는다
  (프리매큐어 추상화 방지).

### 2. 원이 시간 초과로 사라지는지 여부

- **선택지**: (a) 원이 일정 시간 동안 탭되지 않으면 자동으로 사라지고 새 원이 나타남(오스감형
  에임 트레이너), (b) 원이 탭될 때까지 계속 그 자리에 남아있음
- **결정**: (b) 계속 남아있음, 시간제한 없음
- **이유**: v1 스펙에 "판정: 원 안을 탭하면 명중 / 밖이면 미스"라고만 되어 있고 원의 자동 만료
  시간은 어디에도 없다. 임의의 수치(예: 800ms)를 새로 만들어 넣기보다는, 명시된 규칙만 정확히
  구현하고 30초 전체 제한 + 레벨업에 따른 원 축소만으로 난이도 곡선을 만드는 쪽을 택했다. 이후
  실제로 플레이해보고 "너무 쉽다"는 피드백이 나오면 그때 자동 만료를 추가하는 게, 근거 없는
  수치를 먼저 넣는 것보다 안전하다.

### 3. "빠를수록 보너스" 점수 공식

- **선택지**: v1에 구체적 공식이 없어 새로 설계해야 했다.
- **결정**: `점수 = 100(기본) + max(0, round((600 - 반응시간ms) / 600 * 100))`
  — 즉 원이 나타난 순간 바로 맞히면 +200점(기본 100 + 보너스 100), 600ms 이후에 맞히면
  보너스 없이 100점만 획득. 600ms~0ms 구간은 선형 보간.
- **이유**: 600ms는 일반적인 성인의 시각 자극 반응시간(약 200~300ms)에 조준 이동 시간을 더한
  값으로, 너무 후하지도(항상 만점) 너무 박하지도(거의 보너스 없음) 않은 중간값이다. 텐텐오락실
  오마주 후보와 달리 이 게임은 v1 유래 게임이라 특정 경쟁 앱의 수치를 피해야 할 이유는 없지만,
  검증 가능한 근거(일반 반응시간 범위)에 기반한 자체 수치를 새로 정했다.
- **안전장치**로 `오마주 안전장치` 섹션에서 언급한 "룰 수치는 자체 정의" 원칙과 별개로, 이
  게임은 v1 유래(텐텐오락실과 무관)이므로 그 규칙이 적용되는 대상은 아니다.

### 4. 레벨/원 크기 축소 곡선

- **선택지**: (a) 명중할 때마다 1레벨씩 축소, (b) N회 명중마다 1레벨씩 축소
- **결정**: (a) 명중 1회 = 1레벨, 반지름 5px씩 축소, 80px → 30px (총 11레벨, 10회 명중 후
  최소 크기 고정)
- **이유**: 30초 안에 보통 15~30회 정도 명중이 가능한 게임 길이를 가정하면, 10회 만에 최소
  크기에 도달해 남은 절반 이상의 시간을 "가장 어려운 상태"로 플레이하게 된다. N회마다 축소하는
  방식은 몇 회인지에 대한 근거 없는 숫자를 하나 더 만드는 것이라 피했다.

### 5. 미스에 대한 페널티

- **결정**: 점수 감점 없음, 미스 횟수만 별도로 집계해 결과 화면의 정확도(%) 계산에 사용
- **이유**: v1 스펙이 "명중 +100"만 언급하고 미스 페널티는 언급하지 않는다. 감점을 넣으면
  실수로 빗나간 탭 하나가 두 배로 손해(놓친 기회 + 감점)가 되어 체감 난이도가 크게 올라가므로,
  스펙에 없는 페널티를 임의로 추가하지 않았다.

### 6. 시각 연출 수준

- **결정**: 원에 그라디언트 + 은은한 펄스(숨쉬기) 애니메이션, 명중 시 위로 떠오르며 사라지는
  "+점수" 팝업, 미스 시 화면 테두리 빨간 플래시. 사운드는 추가하지 않음.
- **이유**: "프로덕트 레벨만큼 꼼꼼하게"라는 요청에 맞춰 기존 게임들(`dice`의 결과 이미지
  그라디언트·애니메이션, `race`의 이펙트)과 비슷한 완성도를 목표로 했다. 사운드는 저장소에
  오디오 에셋이 전혀 없고(`public/assets`에 이미지/GLB만 존재) 새 오디오 파일을 추가하는 건
  이번 스코프를 벗어난다고 판단해 제외했다.

### 7. 최고점수 저장 키

- **결정**: `src/shared/score-store.ts`를 신설해 `rhh_aim-trainer_best` 키로 저장 —
  v2 계획서에서 이미 정한 네이밍 규칙(`rhh_<game>_best`)을 그대로 따름. 추가 결정 없음.
