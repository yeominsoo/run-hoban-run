# 합이 10 퍼즐(sum-ten-puzzle) 구현 노트 (2026-07-16)

텐텐오락실 오마주 후보 10종(Phase 4) 중 일곱 번째이자 Phase 4 "동시 랭킹형" 계열의
마지막 게임. 계획서에서 유일하게 "로컬 연습 모드 + WS 방 모드 선택"으로 지정된 게임이라
지금까지의 6종과 달리 서버 유무를 모두 지원하는 하이브리드 구조로 만들었다.

## 스펙에 없어 직접 정한 수치와 근거 (오마주 안전장치 적용)

계획서의 "오마주 안전장치" 원칙("룰 수치도 자체 정의 — 제한시간·난이도 곡선 등은 원조와
동일하게 맞추지 않는다")에 따라 실제 참고 대상인 "사과게임"과 의도적으로 다르게 잡았다.

- **격자 8행×10열(80칸)**: 사과게임의 17×10(170칸)보다 작게 — 모바일 화면 한 판에
  스크롤 없이 다 들어오는 크기를 우선했다.
- **라운드 90초**: 사과게임의 120초보다 짧게.
- **점수 = 지운 칸 수 누적**: 사각형 하나를 지울 때마다 그 안에 포함된 칸 수만큼
  점수를 준다 — 칸이 많을수록(더 어려운 조합일수록) 점수도 커지도록 설계했다.

## 로컬 모드와 WS 모드가 그리드 렌더링/드래그 로직을 공유하는 구조

같은 캔버스, 같은 `renderGrid()`/드래그 선택 로직을 두 모드가 그대로 공유한다 —
차이는 오직 "격자를 누가 만드는지"(로컬 모드: 클라이언트가 즉석에서 생성, WS 모드:
서버가 만들어 `round_start`로 배포)와 "클리어를 어디에 알리는지"(로컬 모드: 아무 데도
안 알림, WS 모드: 서버에 `clear` 메시지 전송)뿐이다. 덕분에 그리드 렌더링 버그를 한
곳에서만 고치면 두 모드 모두에 적용된다.

## 안티치트 — "공개 정보"와 "조작 가능한 점수"를 구분

이 게임은 앞선 6종과 달리 격자 자체(공개 정보)가 아니라 **점수 자기 보고**가 공격
표면이다. 클라이언트가 실제로 합이 10인 칸만 지우는지는 클라이언트 코드를 믿을 수밖에
없지만, 조작된 클라이언트가 "지우지도 않은 칸을 지웠다"고 우기는 것은 막아야 한다. 그래서
서버가 방 생성 시 만든 `initialGrid`를 플레이어별로 복사해 독립적으로 추적하고
(`player.board`), 클라이언트의 `clear` 요청을 **그 사본 기준으로** 재검증한다 —
이미 지워진 칸을 다시 지우려 하거나 합이 10이 아니면 `ok:false`로 거부하고 점수를 주지
않는다. 클라이언트는 반응성을 위해 자기 판단으로 먼저 지우고 점수를 올리지만
(낙관적 업데이트), 라운드 종료 시 최종 순위는 서버가 검증한 점수를 기준으로 삼는다.

## 검증

- **raw WebSocket 스크립트**: 서버가 브로드캐스트한 격자에서 실제로 합이 10인 인접
  쌍을 찾아 `clear` 메시지로 제출해 정상 처리(`ok:true`, 점수 반영)를 확인. 같은 칸을
  다시 지우려는 시도와, 단일 칸(합이 10이 될 수 없음)을 제출하는 시도 모두 `ok:false`로
  거부됨을 확인해 안티치트 검증을 마쳤다.
- **Playwright UI 테스트(솔로 모드)**: 모바일 세로(390×844)에서 "혼자 연습하기"를
  선택해 격자를 렌더링하고, 테스트 전용 상태 노출(`#test-state`의 `data-grid`/
  `data-cellSize`)로 실제 합이 10인 인접 쌍의 픽셀 좌표를 계산해 마우스
  드래그(`mouse.move`→`down`→`move`→`up`)로 클리어, 점수 반영과 캔버스 상 셀이
  실제로 사라짐을 스크린샷으로 확인.
- **Playwright UI 테스트(방 모드)**: 호스트+게스트 2탭으로 로비 → 카운트다운 → 플레이
  화면까지 진행해 동일한 방식으로 드래그 클리어 후, 실시간 보드에 점수가 반영되고
  다른 플레이어(게스트)는 0개로 정상 표시됨을 확인.

## 배포

WAS 배포는 앞선 게임들과 동일한 절차(`ws-server/README.md`)를 따른다. 로컬 연습
모드는 서버가 필요 없으므로 WAS 반영 전에도 Firebase 배포만으로 바로 사용 가능하다.
