# 선공/후공 결정전(가위바위보 + 팀 주사위 대표선출) 공용 패턴 (2026-07-16)

## 배경

사용자 요청: "선공 후공이 존재하는 게임에서는 반드시 상대와 가위바위보를 하도록 해줘.
팀전의 경우 팀 내에서 주사위를 굴려서 가장 높게 나온 사람이 가위바위보 하도록 해줘.
가위바위보는 단판승으로."

기존에는 리버시/오목이 "방을 만든 사람(호스트)이 항상 흑(선공)"이었고, 윷놀이/전략윷놀이는
"먼저 접속한 사람 순서대로 `turnOrder`/`moveOrder` 배열을 그대로 사용"해 사실상 입장
순서가 곧 선공 순서였다. 둘 다 실력이나 운과 무관하게 "누가 먼저 방에 들어왔는가"로
유불리가 갈리는 구조라, 이를 가위바위보(필요시 팀 주사위 대표선출)로 대체했다.

## 대상 게임 판별

선공/후공(턴 순서)이 실제로 존재하는 게임만 대상으로 삼았다:

- **리버시(reversi), 오목(gomoku)** — 1:1 턴제, 흑/백 선후공이 승부에 영향을 준다.
- **윷놀이(yutnori)** — 2~4인, 말을 순서대로 이동. 4인이면 2:2 팀.
- **전략윷놀이(strategy-yutnori)** — 항상 4인 2:2 팀 고정.

두더지 사냥·순서 기억 챌린지·구구단 스퍼트 등 나머지 WS 게임들은 전원이 "동시에" 행동하는
구조라 선공/후공 개념 자체가 없어 대상에서 제외했다. 줄다리기 배틀·영역 쟁탈전도 1:1이지만
매 프레임 동시에 입력하는 실시간 게임이라 "누가 먼저 두는가"가 존재하지 않는다.

## 공용 순수 로직 — `ws-server/starting-order.mjs`

- `resolveRps(choiceA, choiceB)`: 가위바위보 승자 판정. 비기면 `null`(재도전 필요).
- `rollDiceOff(tokens, rng)`: 팀(2인 이상) 내부에서 주사위로 대표 1명을 뽑는다. 동점이면
  동점자끼리만 재굴림(전원 재굴림이 아니다). 팀원이 1명이면 굴릴 필요 없이 그 사람이
  바로 대표(`rounds: []`). 반환값의 `rounds`는 애니메이션용 전체 굴림 기록(재굴림
  포함), `winnerToken`은 최종 대표.

각 게임의 room 상태 관리(타이머, 소켓, 재접속)는 게임마다 다르게 생겼기 때문에 공용
모듈에 넣지 않고, 판정 계산만 공유했다 — `yutnori-rules.mjs`(순수 로직)와
`yutnori.mjs`(방/소켓 오케스트레이션)를 나눈 이 저장소의 기존 관례와 같은 원칙이다.

## 진행 흐름

1. **1:1 게임(리버시/오목)**: 방장이 만들고 상대가 들어와 카운트다운이 끝나면, 곧바로
   `deciding` 페이즈로 들어가 두 플레이어가 가위바위보를 낸다. 팀이 없으므로 주사위
   단계는 없다.
2. **팀 게임(윷놀이 3~4인, 전략윷놀이 4인 고정)**: 호스트가 "게임 시작"을 누르면
   서버가 팀(`buildYutTeams`/고정 2:2)을 구성하고, 팀원이 2명인 쪽만 주사위 대표선출을
   진행한다(팀원이 1명이면 그 사람이 바로 대표 — 윷놀이 2인전이 이 경우). 양 팀 대표가
   정해지면 대표끼리 가위바위보로 넘어간다.
3. **가위바위보는 단판**: 비기면 즉시 재도전(같은 라운드를 새로 하는 것이지 "다음
   판"으로 넘어가는 게 아니다) — `decide_tie` 브로드캐스트 후 900ms 뒤 다시 선택
   가능. 응답 제한시간(8초)을 넘기면 서버가 무작위로 대신 낸다.
4. 승자가 정해지면 `decide_result`로 알리고, 1.8초 뒤 실제 게임이 시작된다. 이때
   승자(또는 승자 팀의 대표)가 정확히 첫 턴을 갖도록 `turnIndex`/`moveIndex`를
   승자 토큰의 위치로 맞춘다 — 나머지 턴 순환 순서 자체는 기존 로직(참가자 배열
   순서를 그대로 순환)을 건드리지 않고, 시작 지점만 승자로 옮긴다.

## 리버시/오목: 호스트가 항상 흑이 아니게 됨

기존에는 `colorOf`/`tokenOfColor`가 `room.players[0]`(호스트)을 흑으로 고정했는데, 이제
`room.blackToken`/`room.whiteToken`을 가위바위보 결과로 채워 넣고 그걸 기준으로
판정하도록 바꿨다. 로비 안내 문구도 "방을 만든 쪽이 흑(선공)"에서 "가위바위보(단판)로
흑(선공)을 정해요"로 수정했다.

## 윷놀이/전략윷놀이: 팀 구성은 그대로, 시작 인덱스만 이동

`buildYutTeams`(윷놀이)와 고정 2:2(전략윷놀이)의 팀 나누는 방식 자체는 손대지 않았다 —
`tokens` 배열의 순서(참가자 입장 순서)로 팀을 짜는 기존 로직을 그대로 쓰고, 가위바위보
승자의 `turnOrder`/`moveOrder` 상 위치를 찾아 `turnIndex`/`moveIndex`를 그 값으로
설정하는 것만 추가했다. 턴이 배열을 순환하는 방식 자체는 기존과 동일해서, "누가
먼저 시작하는가"만 바뀌고 그 이후 순서 로직은 회귀 위험 없이 그대로 유지된다.

## 팀에서 대표가 이탈하면?

윷놀이/전략윷놀이 모두 `deciding` 페이즈 도중 누군가 나가면(재접속 유예 없이 완전히
나가는 경우) 결정전을 취소하고 로비로 돌려보낸다 — `decide_aborted` 브로드캐스트로
클라이언트를 강제로 로비 화면으로 전환시키고, 호스트가 다시 "게임 시작"을 누르면
처음부터 다시 결정전이 진행된다. 4인 고정인 전략윷놀이는 애초에 한 명이라도 빠지면
게임 자체를 진행할 수 없어 이 처리가 특히 중요하다.

## 클라이언트 UI — 공용 CSS `src/shared/starting-order.css`

가위바위보 손 아이콘(`src/shared/hand-icons.ts`, 원래 `src/pages/rps/hand-icons.ts`에
있던 걸 4개 게임이 공유하도록 이동)과 주사위 대표선출 카드(`.dice-off-player`,
`.dice-off-face`) 스타일을 공용화했다. 팀 주사위 대표선출은 서버가 재굴림까지 포함한
전체 기록(`rounds`)을 한 번에 계산해서 보내주고, 클라이언트가 그 기록을 700ms 간격으로
순차 재생하는 방식으로 애니메이션한다 — 서버는 실제 판정 타이밍만 책임지고(재도전
가능 여부, 최종 대표 확정), 애니메이션 연출은 순수하게 클라이언트가 담당해 서버·클라이언트
타이밍이 어긋나도 게임 결과에는 영향이 없다.

## 검증

- **리버시/오목**: raw WebSocket 스크립트로 "비김 → 재도전 → 결판" 흐름과, 승자가
  정확히 흑(선공)으로 배정되는지 확인. Playwright로 모바일 세로(390×844)에서 실제
  UI 흐름(가위바위보 버튼 클릭 → 결과 화면 → 대국 시작)도 확인했다.
- **윷놀이**: 2인전(팀원 1명씩, 주사위 없이 바로 가위바위보)과 4인전(2:2, 양 팀 다
  주사위 대표선출 필요, 그중 한 팀은 동점이 나서 재굴림까지 발생)을 모두 raw
  WebSocket으로 확인 — 가위바위보 승자가 정확히 `game_update.currentTurnToken`으로
  잡히는지까지 검증했다. Playwright로 2인전 UI 흐름도 확인.
- **전략윷놀이**: 4인 고정, raw WebSocket으로 양 팀 주사위 대표선출(한쪽은 재굴림
  포함) + 대표끼리 가위바위보(비김 재도전 1회 포함)까지 확인, 승자가 `moveOrder`상
  정확히 첫 무버로 배정되는지 검증. Playwright로 4개 브라우저 컨텍스트를 동시에 띄워
  실제 UI로도 로비 → 결정전 → 대국 시작까지 확인했다.

## 배포

리버시/오목/윷놀이/전략윷놀이 4개 게임 모두 `ws-server/*.mjs` 서버 코드가 바뀌었으므로
WAS(운영 서버) 앱 컨테이너 재빌드·재기동이 필요하다(`ws-server/README.md` 절차 참고).
프론트엔드만 바뀐 변경이 아니라는 점에서 이전의 "로비 팁 카드 추가"류 변경과 다르다.
