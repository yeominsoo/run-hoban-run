# 타이핑 생존 구현 노트 (2026-07-13)

## 원본 스펙 (v1)

- 화면 상단에서 떨어지는 단어를 바닥에 닿기 전에 타이핑
- 단어 소스: 내장 단어 목록(영어 100개 + 한국어 100개)
- 레벨: 10초마다 속도 업, 동시 단어 수 증가
- 종료 조건: 단어가 바닥에 닿으면 HP -1 (HP 3)
- 점수: 단어 제거 수 × 레벨 배수
- Canvas(단어 애니메이션) + DOM input(입력 필드)
- 모바일: `visualViewport` 기준 높이 재계산, 모바일은 영어 단어 전용 모드 권장

## 결정과 이유

1. **매칭 방식**: 입력 필드에 키를 입력할 때마다(각 `input` 이벤트마다) 현재 입력값과
   화면에 떠 있는 모든 단어를 대소문자 무시하고 정확히 비교한다. 일치하는 단어가 있으면
   즉시 제거하고 입력창을 비운다. Enter나 별도 제출 버튼을 요구하지 않는다 — 타이핑
   생존류 게임(ZType 등)의 표준 UX가 실시간 매칭이고, 제출 버튼을 추가하면 오히려
   "생존"의 긴박감이 떨어진다.
2. **동일 단어 중복 시 우선순위**: 화면에 같은 텍스트의 단어가 두 개 이상 떠 있으면
   바닥에 더 가까운(=더 급한) 단어를 제거한다.
3. **속도/동시 단어 수 증가 곡선**: 10초마다 낙하 속도 ×1.12, 최대 동시 단어 수 +1(최대
   6개까지). 초기값은 속도 40px/s, 최대 동시 단어 1개로 시작 — 스펙에 구체 수치가 없어서
   볼 피하기의 "30초마다 ×1.2" 패턴보다 살짝 완만한 배율을 골랐다. 이 게임은 조건 충족
   시(레벨 상승) 동시에 화면의 단어 개수까지 늘어나 체감 난이도가 속도 하나만 오르는
   게임보다 훨씬 가파르게 오르기 때문에, 배율 자체는 더 보수적으로 잡았다.
4. **단어 언어 비율**: 매 스폰마다 영어/한국어 풀에서 50:50으로 무작위 선택한다(모바일
   제외, 아래 6번 참고). 한쪽으로 치우치면 "영어 100개 + 한국어 100개"를 준비한 의미가
   없어진다.
5. **점수 공식**: "단어 제거 수 × 레벨 배수"를 그대로 따라, 단어를 제거할 때마다
   `현재 레벨`만큼 점수를 더한다(레벨 = 10초 경과 단계 수 + 1). 레벨이 오를수록 단어 하나당
   가치가 커져, 뒷심을 발휘할수록 보상이 커지는 구조.
6. **모바일 언어 선택은 접속 로캘 기준(사용자 지시로 변경)**: 최초 구현은 "모바일이면
   무조건 영어"였는데, 이후 사용자가 "모바일에서는 접속 로캘 기준으로 한국이면 한국어,
   아니면 영어"로 바꿔달라고 요청했다. `window.matchMedia('(pointer: coarse)')`로 모바일
   여부를, `navigator.language`가 `ko`로 시작하는지로 로캘을 판별해 모바일에서는 그 결과에
   따라 단일 언어 풀만 스폰한다(데스크톱은 기존대로 영어+한국어 50:50 유지). 모바일 소프트
   키보드에서 언어를 오갈 필요가 없어지는 게 핵심 — 기기 로캘이 이미 사용자가 어느 쪽
   키보드를 켜놨을지에 대한 가장 신뢰할 수 있는 신호다.

### 사전 기반 대규모 단어 목록으로 교체 (사용자 지시)

처음엔 손으로 고른 영어 100개 + 한국어 100개였는데, 사용자가 "국어사전에서 두 글자 이상
단어를 모두 추출해 쓸 수 있게 만들어달라"고 요청해 실제 오픈소스 사전에서 추출한 대규모
목록으로 교체했다.

- **한국어 소스**: 이 개발 환경에 `apt-get install hunspell-ko`로 설치한
  `/usr/share/hunspell/ko.dic`(spellcheck-ko 프로젝트, MPL-1.1/GPL-2+/LGPL-2.1+ 중
  선택 가능한 라이선스, 저작권자 Changwoo Ryu 외 기여자). "표준국어대사전"(국립국어원)은
  API 키가 있어야 접근 가능해 이 세션에서 쓸 수 없었고, 대신 널리 쓰이는 오픈소스 한국어
  맞춤법 사전을 "국어사전"의 실질적 대체재로 썼다.
  - **인코딩 함정**: `.dic` 파일이 한글을 완성형(NFC, 가/나/다처럼 한 글자=한 코드포인트)이
    아니라 자모 분해형(NFD, ㄱ+ㅏ처럼 낱자로 분리된 형태)으로 저장하고 있어서, 처음에
    `[가-힣]` 정규식으로 필터링했더니 매칭이 0건이었다. `unicodedata.normalize('NFC', ...)`로
    정규화한 뒤에야 정상적으로 필터링됐다 — 시스템 사전 파일을 다룰 때 완성형/분해형 인코딩
    차이를 항상 의심할 것.
  - `/FLAGS` 접미사(hunspell 활용형 클래스 참조)를 제거하고, 순수 한글 음절 2~7자인 항목만
    남겨 98,703개를 추출했다. 활용형(예: "가감하다"/"가감하여"/"가감한")이 개별 항목으로도
    실려 있어 그대로 포함했다 — 전부 실제로 쓰이는 유효한 한국어 단어라 사용자의 "모두
    추출" 지시를 문자 그대로 따랐다.
- **영어 소스**: `apt-get install wamerican`으로 설치한 `/usr/share/dict/words`
  (SCOWL, Kevin Atkinson, 자유 재배포 라이선스)에서 소문자 2~10자 알파벳 단어만 필터링해
  52,383개를 추출했다. 사용자는 한국어 사전만 언급했지만, 한국어만 9만 개대로 불려놓고
  영어는 예전 100개 그대로 두면 두 언어 간 단어 다양성이 극단적으로 불균형해진다고 판단해,
  같은 논리(시스템에 이미 있는 오픈소스 사전 전체 추출)를 영어에도 동일하게 적용했다.
- **저장 형식**: 거대한 배열을 `.ts` 리터럴로 두면 diff·타입체크 부담이 커서, `words-ko.json`
  /`words-en.json`으로 분리하고 `word-lists.ts`에서 JSON import로 재노출했다
  (`tsconfig.json`에 `resolveJsonModule: true` 추가). Vite의 페이지별 코드 스플리팅
  덕분에 이 데이터는 `/typing-survival/` 방문 시에만 내려받는다(다른 페이지 번들에는
  영향 없음).
- **번들 크기 트레이드오프**: gzip 기준 한국어 약 310KB + 영어 약 139KB, 합쳐서 약 450KB가
  이 페이지 전용으로 추가된다. 다른 페이지(보통 2~10KB대)보다 훨씬 크지만, "사전 전체 추출"
  요청 자체가 큰 데이터셋을 의미하고 페이지별로 격리돼 있어 감수할 만하다고 판단했다. 이후
  실제 체감 로딩이 문제라면 최초 방문 시 지연 로딩(동적 import)하는 식으로 개선할 수 있다.
- **난이도 필터링 안 함**: 사전에서 추출한 9만 개 이상의 단어에는 일상적이지 않은 전문
  용어·고어도 섞여 있을 수 있다. 빈도 사전이 없어 "쉬운 단어"만 고를 근거가 없었고,
  사용자가 "모두 추출"이라고 명시했기 때문에 난이도 기준 축소는 하지 않았다.
7. **`visualViewport` 대응**: 소프트 키보드가 열리면 `window.visualViewport.height`가
   줄어드는데, 이 값을 캔버스 높이 계산에 반영해 키보드에 가려지지 않는 영역에서만 단어가
   떨어지도록 한다. `visualViewport`를 지원하지 않는 환경(구형 브라우저)에서는
   `window.innerHeight`로 자연스럽게 폴백한다.
8. **입력 포커스 유지**: 게임 중 입력 필드가 blur되면(예: 실수로 다른 곳 탭) 즉시 다시
   포커스를 맞춘다 — 안 그러면 키 입력이 아예 씹혀서 억울하게 단어를 놓치게 된다.

## 실기기 피드백으로 발견한 버그: 키보드가 뜨면 상단이 안 보임

`visualViewport` 기반으로 캔버스 높이만 줄이면 될 거라 예상했지만, 실제 모바일 기기에서
키보드가 열리면 브라우저가 "포커스된 입력 요소를 보이게 하려고 문서 전체를 위로
스크롤"해버리는 사례가 있었다 — 이 경우 캔버스 크기는 그대로인데 문서 자체가 스크롤돼
화면 상단(단어가 떨어지는 영역)이 뷰포트 밖으로 밀려나 아무 단어도 안 보이는 문제가 생겼다.
`resize` 이벤트만 감지해 캔버스 내부 크기를 조정하는 것으로는 이 "문서 레벨 스크롤"을 막을
수 없었다. 해결책:

- `body`를 `position: fixed; inset: 0`으로 고정해, 브라우저의 "포커스 요소로 자동 스크롤"
  동작이 애초에 먹히지 않게 만들었다(고정 위치 문서는 브라우저가 스크롤시킬 대상이 없다).
- `visualViewport`의 `resize`/`scroll` 이벤트 양쪽을 모두 구독해서, 크기가 바뀌거나 문서가
  조금이라도 스크롤되면 즉시 `window.scrollTo(0, 0)`으로 되돌리고 `html`/`body` 높이를
  `visualViewport.height`에 맞춰 명시적으로 다시 설정한다.

**한계**: Playwright 헤드리스 브라우저는 실제 소프트 키보드를 띄우지 않기 때문에 이 수정을
자동 E2E 테스트로 재현·검증할 수 없다. 코드 리뷰와 정적 분석 수준으로 확인했고, 실기기
검증은 사용자 피드백에 의존했다.
