# 순서 기억 챌린지(memory-sequence) 구현 노트 (2026-07-15)

텐텐오락실 오마주 후보 10종(Phase 4) 중 두 번째 게임. 두더지 사냥과 달리 "동시 랭킹형"이
아니라 **탈락 서바이벌형**(Simon Says류)이라, 라운드 구조와 승자 판정 로직을 새로 설계했다.

## 확정한 수치

- 방 인원: 2~8명 (두더지 사냥과 동일한 정책 유지)
- 타일 4색, 라운드 1은 시퀀스 길이 3에서 시작해 라운드마다 1칸씩 늘어남(누적 재생 방식 —
  라운드마다 처음부터 전체 시퀀스를 다시 보여준다. 매 라운드 새 타일만 보여주는 방식보다
  어렵지만 이 장르의 표준적인 난이도 곡선이다)
- 타일 노출 550ms, 간격 220ms, 라운드 시작 후 첫 타일까지 700ms 준비 시간
- 입력 제한시간 = 시퀀스 길이 × 1800ms + 4000ms 버퍼
- 라운드마다 시퀀스를 못 맞춘 사람은 즉시 그 라운드에서 탈락(게임에서 제외), 마지막 생존자가
  우승

## 서버 권위 판정 — 시퀀스를 "보여주되 한 번에 넘기지 않는다"

가장 중요한 설계 결정: 서버가 전체 시퀀스 배열을 한 메시지로 보내지 않는다. 만약 그렇게
하면 네트워크 탭만 열어봐도 답을 알 수 있어 기억력 게임의 핵심이 무너진다. 대신 `tile_reveal`
이벤트를 타일 하나당 하나씩, 서버가 `setTimeout`으로 실제 시간 간격을 두고 순차 브로드캐스트한다
— 치팅을 원천 차단하진 못하지만(콘솔 로그를 실시간으로 지켜보면 결국 사람이 보는 것과 동일한
속도로만 알 수 있다), 최소한 "미리 다 알고 시작"하는 것은 불가능하게 만든다. 이건 라이어게임의
"필드 자체를 payload에 만들지 않는다"는 서버 권위 원칙을 실시간 게임에 맞게 응용한 것이다.

## 동시 전멸 타이브레이크

한 라운드에서 생존자 전원이 동시에 탈락하면(생존자 1명이 남아있던 상태에서 마지막 한 명마저
틀리는 경우) 우승자가 없어져 버리는 문제가 있다. 이번 라운드에 "틀리기 전까지 맞춘 타일 수"가
가장 많은 사람을 우승으로 판정하도록 했다(동률이면 공동 우승 — 두더지 사냥과 동일한 정책).
raw WebSocket 시나리오 테스트로 실제로 두 명이 서로 다른 진행도에서 탈락했을 때 더 깊이
맞춘 쪽이 정확히 우승 판정되는지 확인했다.

## 구현 중 발견한 버그

첫 구현에서 `startRound()`가 매 라운드 무조건 타일을 1개씩만 추가해서, 라운드 1이 의도한
길이 3이 아니라 길이 1로 시작하는 버그가 있었다(`START_LENGTH` 상수를 정의만 하고 실제로
쓰지 않음). `round_start`로 실제 `sequenceLength`를 raw WS 테스트로 확인하는 과정에서 바로
드러났다 — "목표 길이까지 채워질 때까지 push" 방식으로 고쳤다(`room.sequence.length === 0
? START_LENGTH : room.sequence.length + 1`을 목표로 삼는 while 루프).

## 검증

로컬(`PORT=8799`)에서 raw WebSocket 스크립트로 4가지 시나리오를 확인:
1. 2명, 한 명은 정답을 그대로 따라 탭, 한 명은 고의 오답 → 오답자만 탈락, 남은 생존자 즉시 우승.
2. 3명, 라운드1 전원 통과(시퀀스 길이 3→4로 증가) → 라운드2에서 2명 탈락 → 1명 우승,
   `finalRound`가 정확히 2로 기록됨.
3. 2명, 서로 다른 진행도에서 동시 탈락 → 진행도가 더 깊었던 쪽이 타이브레이크로 우승.
4. 게임 도중 `leave` → 남은 플레이어 즉시 승리, 소켓만 끊고 `rejoin` → 진행 중이던
   라운드/생존자 상태 복원.

WAS 배포는 두더지 사냥과 동일한 절차(`ws-server/README.md`)를 따랐다.
