# 줄다리기 배틀(tug-of-war-battle) 구현 노트 (2026-07-16)

텐텐오락실 오마주 후보 10종 중 여덟 번째이자 Phase 5 "실시간 조작 동기화형" 계열의 첫
게임. Phase 4의 7종("동시 랭킹형" — 라운드 시작 시 문제/격자를 배포하고 각자 진행)과
근본적으로 다른 뼈대가 필요했다 — 이 게임은 **매 탭마다 실시간으로 상태가 바뀌고 양쪽에
계속 브로드캐스트**돼야 한다.

## 왜 1:1 전용인가 — `rps`의 1v1 모드를 그대로 참고

계획서 명세("1:1 실시간 연타 대결")를 그대로 따라 Phase 4의 "방 만들기(인원수 선택) → 로비
대기 → 호스트가 게임 시작" 패턴을 쓰지 않고, 이 저장소에 이미 있던 `rps`의 1v1 모드 패턴을
가져왔다 — 방을 만들면 곧바로 대기 화면이 뜨고, 두 번째 사람이 join하는 순간 자동으로
매칭되어 카운트다운 후 시작한다. 인원수 선택 UI 자체가 없다(항상 2명).

## 스펙에 없어 직접 정한 수치와 근거

- **판정 축 -100~100, 탭당 2씩 이동**: 중앙(0)에서 한쪽 끝까지 약 50번의 탭이 필요하다 —
  "연타 대결"이라는 느낌이 나면서도 1~2초 만에 끝나버리지 않을 정도의 지속시간을 노렸다.
- **탭 레이트리밋 35ms(초당 약 28회)**: 사람이 실제로 낼 수 있는 최고 연타 속도(초당
  10~15회 내외)보다 넉넉히 높게 잡아서 정상적인 빠른 연타는 전혀 막지 않으면서도, 매크로나
  자동 클릭 스크립트가 초당 수백 회씩 보내는 극단적인 경우만 걸러낸다.
- **제한시간 30초, 시간 초과 시 더 당긴 쪽 승리**: 두 사람의 실력차가 크지 않으면 계속
  0 근처에서 밀고 당기기만 할 수 있어 최소한의 안전장치로 넣었다.

## 서버 권위 원칙 — "판정 축은 서버만 갖는다"

`position`은 오직 서버에만 존재하는 단일 진실 값이다. 클라이언트는 `tap` 메시지만 보내고,
실제로 줄이 얼마나 움직였는지는 서버의 `position_update` 브로드캐스트로만 안다 — 클라이언트가
자기 화면에서 먼저 낙관적으로 마커를 움직이지 않는다(합이 10 퍼즐의 점수와 달리, 이
게임은 매 프레임 상대와 실시간으로 비교되는 값이라 낙관적 업데이트가 어긋나면 두 화면이
서로 다른 걸 보여주는 훨씬 심각한 문제가 된다). 탭 레이트리밋도 클라이언트가 아니라
서버에서 걸어 클라이언트를 신뢰하지 않는다 — 할리갈리 구현 때 확립한 "서버 수신 순서
기반 판정" 원칙을 그대로 이어받았다.

## 검증

- **raw WebSocket 스크립트**: 호스트+게스트 2개 소켓으로 방 생성 → 참가 즉시 자동 매칭
  → 카운트다운 → 대결 시작까지 확인. 호스트가 `tap`을 반복 전송해 `position`이 -100에
  도달할 때까지의 전체 시퀀스와, 레이트리밋에 걸려 일부 탭이 무시되는 것(각 tap 이후
  `position_update`를 기다리되 타임아웃되면 무시된 것으로 처리)까지 확인. 최종
  `game_over`의 승자가 실제로 먼저 도달한 쪽과 일치함을 검증.
- **Playwright UI 테스트**: 모바일 세로(390×844)에서 호스트가 방을 만들고 게스트가
  참가하면 곧바로 카운트다운이 뜨는지, 큰 "당기기!" 버튼을 15회 연속 클릭했을 때 줄
  마커가 실제로 호스트 쪽으로 이동하는지(중앙 50% → 35%) 스크린샷과 DOM 상태로 확인.
