```text
너는 `run-hoban-run` 저장소를 분석하고, 현재 그래픽 품질 문제를 해결하기 위한 기획 및 개발 계획서를 작성하는 시니어 게임/프론트엔드 엔지니어다.

현재 문제 인식:

- 현재 화면은 경마 게임이라기보다 디버그 시뮬레이터처럼 보인다.
- 카메라가 너무 높고 멀어서 말과 기수가 작고 약하게 보인다.
- 주황색 사선 트랙과 과도한 흰 레인 라인이 경마장보다 육상 트랙처럼 보인다.
- 모든 말 위에 이름표가 상시 노출되어 레이스 장면을 덮는다.
- 무료 GLB 말과 앉은 사람 모델의 조합이 자연스러운 경마 기수/말로 보이지 않는다.
- 출발 게이트, 레일, 관중석, 결승 포스트, 거리 표지, 먼지, 접지 그림자 같은 경마 문법이 부족하다.
- Playwright 테스트는 nonblank canvas 정도만 확인하고 있어 시각 품질을 승인하지 못한다.

레퍼런스 해석:

- Rival Stars Horse Racing / Phar Lap:
  - 현실적인 3D 말, 모션 캡처, 중계형 카메라, 최종 직선 카메라가 강점이다.
  - 이 프로젝트에서는 풀 리얼리즘을 따라 하지 말고, "큰 말, 낮은 카메라, 선두/팩 중심 프레이밍"만 가져온다.
- Starters Orders:
  - 그래픽이 낮아도 경마 문법이 있으면 경마처럼 읽힌다.
  - 팩 움직임, 레일, 코스, 중계 정보, 포토피니시/결승 문법을 참고한다.
- Pocket Card Jockey:
  - 리얼리즘 대신 일관된 스타일을 선택한다.
  - 이 프로젝트에는 "스타일라이즈드 방송형 경마"가 가장 적합하다.
- Uma Musume:
  - 속도감, 카메라 컷, 최종 연출의 템포만 참고한다.
  - 캐릭터, 세계관, UI, 명칭, 애니메이션, 음악, 모델을 복제하지 않는다.

권장 최종 방향:

- "스타일라이즈드 방송형 경마"로 잡는다.
- 리얼 GLB 자산 추가보다, 통일된 저폴리/마스코트형 말과 기수 표현을 우선한다.
- 경마장 문법을 먼저 만든다: 타원형 또는 직선 경마 코스, 안쪽/바깥쪽 레일, 잔디/더트 구분, 관중석, 결승 포스트, 출발 게이트, 거리 표지, 먼지, 접지 그림자.
- 카메라는 단일 고각 overview가 아니라 컷 시퀀스로 설계한다:
  - 출발 전 게이트/라인업
  - 초반 사이드 중계 카메라
  - 중반 낮은 3/4 팩 추적 카메라
  - 종반 선두 또는 선택 주자 후방 추적 카메라
  - 결승선 포토피니시 카메라
  - 우승자 발표/위너서클 카메라
- 말 위 이름표는 상시 노출하지 않는다.
  - leaderboard는 이름의 기준 소스로 유지한다.
  - on-canvas callout은 leader, selected runner, skill event, winner 정도에만 제한한다.
- 첫 품질 목표는 모델 고해상도가 아니라 "프레이밍, 달리는 느낌, 접지감, 먼지, 그림자, UI 간섭 제거"다.

계획서 작성 원칙:

- 한국어로 작성한다.
- 구현 코드는 수정하지 않는다.
- 계획서는 `docs/graphics-rework-plan-2026-06-12.md`로 작성한다고 가정하고, 해당 문서의 완성본 내용을 제시한다.
- 기존 `docs/3d-webgame-plan.md`를 무조건 폐기하지 말고, 오늘의 그래픽 리워크 방향과 충돌하는 지점만 명확히 분리한다.
- 실제 결정이 필요한 항목과 권장 기본값을 구분한다.
- "테스트를 위해 제품 DOM/UI/runtime을 바꾸는 것"은 금지한다. 테스트는 사용자-visible 품질을 검증하도록 설계한다.
- 랜덤 무료 GLB를 더 붙이는 방식은 기본 전략에서 제외한다. 외부 에셋을 쓸 경우 horse/rider/rig/animation/license가 맞는 asset pack만 허용한다.
- 헬리콥터 탈락 이벤트는 재미 요소로 유지 가능하지만, 그래픽 리워크 1차 범위에서는 경마 장면의 기본 품질을 해치지 않도록 후순위로 둔다.

계획서에 반드시 포함할 섹션:

1. 개요
   - 왜 그래픽 리워크가 필요한지 5줄 이내로 설명한다.

2. 현재 문제 진단
   - 화면 구도, 카메라, 트랙, 모델, UI, 테스트 기준으로 나눠 설명한다.

3. 목표 비주얼 방향
   - "스타일라이즈드 방송형 경마"를 한 문단으로 정의한다.
   - 좋은 첫 화면/경주 중간/결승 장면이 어떻게 보여야 하는지 구체적으로 설명한다.

4. 레퍼런스 적용 원칙
   - Rival Stars / Phar Lap / Starters Orders / Pocket Card Jockey / Uma Musume에서 가져올 것과 가져오지 않을 것을 표로 정리한다.

5. 비범위
   - 당장 하지 않을 일: 리얼 모션캡처 수준 구현, 무작위 GLB 추가, 우마무스메식 복제, 토너먼트 엔진 재작성, 테스트 편의용 제품 변경 등.

6. UX 및 화면 구성 계획
   - 대형 현장 디스플레이 기준의 기본 레이아웃을 설명한다.
   - 설정 패널, 레이스 캔버스, 리더보드, 콜아웃, 결과 표시의 역할을 나눈다.
   - 말 이름/순위/스킬 정보를 어디에 표시할지 정한다.

7. 3D 장면 리워크 계획
   - 트랙/경마장
   - 말/기수 모델
   - 애니메이션
   - 카메라 컷
   - 이펙트
   - 조명/그림자
   - 모바일/데스크톱 대응

8. 모델 전략
   - A안: 절차형 stylized horse/rider를 직접 구성
   - B안: 라이선스가 맞는 통일 asset pack 사용
   - 각 안의 장단점과 권장안을 정리한다.

9. 개발 단계
   - Phase 0: 현재 상태 캡처와 visual target 정의
   - Phase 1: 카메라/트랙/라벨 제거 중심의 빠른 시각 개선
   - Phase 2: stylized horse/rider 모델과 gallop 애니메이션
   - Phase 3: broadcast camera sequence와 final stretch/finish 연출
   - Phase 4: 우승자 발표와 결과 캡처 품질 개선
   - Phase 5: visual regression 및 Playwright 품질 검증 강화
   - 각 Phase마다 목표, 작업 항목, 산출물, 완료 기준을 적는다.

10. 검증 계획
    - Playwright 스크린샷 검증 기준을 pre-race, mid-race, final-stretch, finish/winner로 나눈다.
    - nonblank canvas 외에 프레이밍, UI 겹침, 콜아웃 개수, 캔버스 점유율, 주요 오브젝트 가시성 기준을 제안한다.
    - 수동 QA 체크리스트도 포함한다.

11. 리스크 및 대응
    - 성능, 에셋 라이선스, 모바일 화면, 모델 애니메이션, 기존 테스트 충돌, 사용자가 기대하는 "웃긴 연출"과 경마 문법의 균형을 다룬다.

12. 우선순위 결론
    - 바로 다음 구현 세션에서 시작할 5개 작업만 정한다.
    - 작업 순서는 "카메라와 UI 간섭 제거 -> 경마장 문법 -> 모델 통일 -> 이펙트/연출 -> 테스트 강화" 원칙을 따른다.
```
