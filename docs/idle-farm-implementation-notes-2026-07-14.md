# 방치형 농장 구현 노트 (2026-07-14)

## 원본 스펙 (v1)

- 목표: 씨앗 심기 → 타이머 → 수확 → 업그레이드 반복
- 입력: 탭(심기/수확), 버튼(업그레이드 구매)
- 작물 종류: 3가지(성장 시간/수익 다름)
- 업그레이드: 수확량 증가, 성장 속도 증가, 자동 수확기
- 저장: `localStorage`에 농장 상태 저장(재진입 시 복구)
- 오프라인 수익: 마지막 접속 시간 기준으로 경과 시간만큼 수익 계산
- Canvas 사용: 아니오(DOM + CSS 애니메이션)
- 모바일 주의사항: 없음

v1 스펙에는 구체적인 수치(작물별 성장 시간/수익, 밭 칸 수, 업그레이드 비용/효과, 랭킹 점수
정의)가 없다. 아래는 그 빈틈을 메운 결정과 근거다.

## 결정과 이유

1. **밭 칸 수: 6칸(2행×3열) 고정, 확장 업그레이드 없음.** 스펙이 언급한 업그레이드는
   "수확량 증가/성장 속도 증가/자동 수확기" 3종뿐이라, 명시되지 않은 "밭 확장" 같은
   기능을 임의로 추가하지 않았다(요청 이상으로 기능을 넓히지 말 것).
2. **작물 3종의 성장 시간/수익을 "느릴수록 효율 좋음"으로 설계**:
   - 당근: 8초 · 4코인(0.50코인/초)
   - 토마토: 25초 · 15코인(0.60코인/초)
   - 수박: 60초 · 45코인(0.75코인/초)
   빠른 작물은 초반 체감 진행감(자주 수확), 느린 작물은 방치 효율(초당 수익 우위)을 담당하게
   해 "심기 선택"에 실제 트레이드오프가 생기도록 했다 — 세 작물의 초당 수익이 전부 같으면
   선택할 이유가 없어진다.
3. **심기 자체는 무료.** 스펙의 입력이 "탭(심기/수확)"뿐이고 씨앗 구매/비용 개념이 없어,
   빈 칸을 탭하면 바로 작물 선택 팝업이 뜨고 코인 소모 없이 심어진다. 대신 아래 업그레이드
   비용 곡선이 "빨리 더 많이 심고 싶으면 업그레이드에 투자"하는 진행 곡선을 담당한다.
4. **업그레이드 비용은 지수 곡선(`round(base * 1.6^level)`)**: 수확량 증가는
   `base=50`, 성장 속도 증가는 `base=60`, 둘 다 레벨 제한 없이 계속 살 수 있지만 비용이
   기하급수적으로 늘어나 자연스럽게 진행 속도를 조절한다(방치형 게임의 표준 패턴). 자동
   수확기는 1회성 구매(500코인)로, 6칸 당근 기준(초당 약 3코인) 약 3분 플레이 지점에
   손이 닿는 첫 번째 큰 이정표가 되도록 잡았다.
   - 수확량 증가: 레벨당 +10% (`multiplier = 1 + level * 0.1`)
   - 성장 속도 증가: 레벨당 -8%, 최저 20%까지(`multiplier = max(0.2, 0.92^level)`) —
     0에 수렴하게 두면 성장 시간이 0이 되어 무의미해지므로 하한을 뒀다.
5. **"성장 중" 판정은 온라인/오프라인을 구분하지 않고 절대 타임스탬프 하나로 통일.**
   각 칸은 `plantedAt`(심은 시각의 절대 `Date.now()` 값)만 저장한다. 진행률·완료 여부는
   항상 `(now - plantedAt) / growMs`로 계산하므로, 앱이 열려 있을 때 500ms마다 도는
   틱이든 페이지를 새로 열 때 딱 한 번 도는 초기화든 완전히 같은 함수
   (`processAutoHarvest(now)` / `plotProgress(plot, now)`)로 처리된다. 별도의
   "마지막 접속 시각(lastSeenAt)" 필드나 "오프라인 전용 계산 분기"를 두지 않았다 — 상태가
   둘로 갈리면 그만큼 버그 표면이 늘어난다.
6. **자동 수확기 없이 방치 후 복귀 = "다 자란 채로 대기"만, 자동 재수확은 없음.** 자동
   수확기를 사지 않은 칸은 `(now - plantedAt) >= growMs`가 되는 순간부터 그냥 "수확
   대기" 상태로 멈춰 있는다(진행률은 100%에서 클램프). 다음 사이클은 플레이어가 직접 탭해
   수확하고 다시 심어야 시작된다 — "자동 수확기"라는 업그레이드가 존재하는 이유 자체가
   이 자동 재시작 여부이기 때문에, 미구매 상태에서 오프라인 동안 몇 사이클이 지났는지 세는
   로직은 만들지 않았다(사면 의미 있어지는 차별점을 오프라인 계산에서도 지켜야 한다).
7. **자동 수확기 보유 시 오프라인 다회 수확 계산**: `cycles = floor((now - plantedAt) /
   effectiveGrowMs)`로 지나간 사이클 수를 구해 `cycles`번치 코인을 한 번에 정산하고,
   `plantedAt += cycles * effectiveGrowMs`로 갱신해 남은 진행분은 계속 이어서 자라게
   한다. 이 계산은 페이지 로드 시 1회, 그리고 자동 수확기 보유 중에는 이후 매 틱마다도
   그대로 재사용된다(6번 결정과 동일한 이유).
8. **오프라인 수확 안내는 "이번 로드에서 처음 계산했을 때, 실제로 코인을 얻었을 때"만
   1회 토스트로 알린다.** 매 500ms 틱마다 `cycles`가 0인 게 대부분이라 토스트를 매번
   띄우면 스팸이 된다 — 인메모리 플래그로 페이지 로드 후 첫 계산에서만 판단한다(이 플래그
   자체는 저장하지 않는다, 새로고침마다 다시 "복귀"로 취급해도 자연스럽다).
9. **랭킹 점수 = 누적 획득 코인(`totalEarned`), 현재 잔액(`coins`)이 아님.** 이 게임은
   다른 7개 게임과 달리 "한 판이 끝나는" 개념이 없어 언제든 랭킹에 등록할 수 있게 만들었다.
   현재 잔액은 업그레이드 구매로 줄어들 수 있어 점수로 쓰면 "돈을 아낀 사람이 유리"해지는
   이상한 지표가 된다 — 평생 벌어들인 총합을 써야 "얼마나 오래·효율적으로 키웠는지"를
   그대로 반영한다.
10. **랭킹 UI 배치를 다른 게임과 다르게 상시 노출.** 기존 7개 게임은 "게임 종료 오버레이"에
    닉네임 입력 폼이 있었지만, 이 게임엔 종료 시점이 없다. 대신 화면 하단에 "내 기록 등록"
    섹션을 상시 배치하고, 헤더의 "누적수익" 배지 옆에 "랭킹보기" 버튼을 뒀다. 코인이 늘어날
    때마다(수확할 때마다) `resetRankingSubmission`을 호출해 저장 버튼을 다시 눌러
    갱신 등록할 수 있게 했다(다른 게임의 "새 판마다 초기화"와 같은 효과를, 여기서는 "점수가
    바뀔 때마다"로 대응).
11. **공용 인프라(`shared/leaderboard.ts`, `shared/game-shell.css`, `shared/score-store.ts`
    는 사용하지 않음) 재사용.** 다만 `best-score`(최고 점수 1회성 기록) 개념이 이 게임에는
    맞지 않아 `score-store.ts`는 쓰지 않고, 헤더의 "누적수익" 배지는 `totalEarned`를
    실시간으로 직접 표시한다.

## 테스트 전략: Playwright `page.clock`으로 실시간 대기 제거

작물 성장 시간이 최대 60초(수박)라 실제 시간을 기다리는 테스트는 느리고 불안정하다.
Playwright 1.60의 `page.clock` API(가짜 타이머)를 페이지 로드 전에 설치해 `fastForward`로
가상 시간을 즉시 흘려보내는 방식으로 테스트했다. `fastForward`는 사이에 있는
`setInterval` 콜백을 전부 실행하는 대신 "한 번만 발동"하는 방식(랩톱 절전모드에서 깨어나는
것과 유사)이라, 정확히 6번 결정에서 설계한 "절대 타임스탬프 기반 계산"이 실시간 틱이든
가짜 시간 점프든 동일하게 맞아떨어지는지도 함께 검증하는 효과가 있었다.

오프라인 수확 계산(7번 결정)은 심은 뒤 시간을 크게 점프시키고 `page.reload()`로 앱을
"재시작"시켜, 저장된 `plantedAt`만으로 여러 사이클치 코인이 한 번에 정산되는지 확인하는
별도 테스트로 검증했다.

## 테스트로 발견한 버그: `beforeunload` 저장이 최신 상태를 덮어씀

처음에는 "닫는 순간까지의 진행을 놓치지 않겠다"는 생각으로 `window.addEventListener
('beforeunload', () => saveState())`를 추가했다. 그런데 `page.goto` → (테스트가
`localStorage`를 직접 덮어써 특정 상태로 세팅) → `page.reload()` 순서로 진행하는 테스트가
전부 코인이 0으로 되돌아가는 문제가 발생했다. 원인은 `page.reload()`가 내부적으로 옛 페이지를
"언로드"하는 과정에서 그 옛 페이지의 `beforeunload` 핸들러가 실행되고, 이 핸들러가 (테스트가
막 덮어쓴 새 값이 아니라) 옛 페이지가 메모리에 들고 있던 오래된 `state`를 다시
`localStorage`에 써버린 것이었다 — 즉 "가장 마지막에 저장한 쪽이 이긴다"는 가정이
깨지는 레이스 컨디션이다.

실제로는 심기/수확/업그레이드 등 상태를 바꾸는 모든 동작이 그 즉시 `saveState()`를
호출하고, 자동 수확기가 있을 때도 500ms 틱마다 저장하므로 `beforeunload`가 커버해야 할
"저장되지 않은 진행"은 최대 500ms 미만의 자동 수확 진행분뿐이다. 그마저도 `plantedAt`
기반 계산(6·7번 결정) 덕분에 다음 로드 때 자연히 다시 계산되어 유실되지 않는다 —
`beforeunload` 핸들러 자체가 득보다 실이 큰 불필요한 안전장치였기에 완전히 제거했다.
**교훈**: "혹시 모르니 닫힐 때도 저장" 같은 방어적 핸들러는, 이미 상태 변경 시점마다
저장하는 설계에서는 새로운 이득 없이 새로운 레이스 컨디션만 만들 수 있다 — 추가하기 전에
"이게 막는 손실이 실제로 존재하는가"를 먼저 따져야 한다.
