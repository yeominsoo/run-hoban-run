# 홀짝 암산(odd-even-math) 구현 노트 (2026-07-16)

텐텐오락실 오마주 후보 10종(Phase 4) 중 다섯 번째 게임. 구구단 스퍼트에서 확립한 "타이머
브로드캐스트 문제 + 첫 정답만 인정 + 정답 개수/평균속도 랭킹" 뼈대를 그대로 재사용하고,
문제 생성과 입력 방식(숫자 입력 → 홀/짝 탭 버튼)만 바꿨다. 계획서에서 예고한 대로 "문제
생성/판정 로직만 바뀌고 나머지는 재사용"이 실제로 잘 통했다.

## 스펙에 없어 직접 정한 수치와 근거

- **피연산자 1~20, 연산자 +/-/×**: 암산으로 결과 홀짝을 즉시 판단할 수 있는 난이도.
  나눗셈은 나머지가 생기면 "결과"의 정의가 애매해져서 처음부터 제외했다.
- **문제 간격 2.5초(구구단 스퍼트는 3초)**: 홀짝 판단은 정확한 곱을 암산하는 것보다
  가볍고 빠른 판단이라 기준시간을 살짝 줄여 템포를 올렸다.
- **입력 방식은 숫자 키패드가 아니라 "홀"/"짝" 두 개의 큰 탭 버튼**: 응답값이 두 가지뿐인
  이진 판단 게임이라 텍스트 입력을 요구하는 건 과한 마찰이다. 모바일에서 오탈자 없이
  한 번의 탭으로 끝나야 "스퍼트"라는 속도감이 산다.

## 서버 골격 재사용 확인

`ws-server/multiplication-sprint.mjs`를 템플릿으로 복사한 뒤 `nextProblem`의 문제 생성부와
`answer` 핸들러의 판정 조건만 바꿔서 완성했다 — 로비/재접속/카운트다운/랭킹 계산 로직은
문자 그대로 동일하다. 계산식(a, b, 연산자)과 홀짝 결과 모두 애초에 비밀일 이유가 없는
공개 정보라, 구구단 스퍼트와 마찬가지로 업다운 넘버 수준의 안티치트 설계(정답 필드
숨기기)가 아예 필요 없다는 점도 동일하게 확인했다.

## 검증

- **raw WebSocket 스크립트**: 호스트가 서버에서 받은 계산식을 그대로 계산해 홀짝 정답을
  5회 연속 제출하며 `correctCount`가 정확히 누적되는지, 고의 오답 제출 시 `correct:false`만
  오고 카운트가 오염되지 않는지 확인.
- **Playwright UI 테스트**: 모바일 세로(390×844)에서 로비 → 카운트다운 → 플레이 화면까지
  실제 UI로 문제 텍스트를 파싱해 홀/짝 버튼을 탭하고, 정답 시 초록 플래시와 버튼
  비활성화(같은 문제 중복 제출 방지)가 정상 동작함을 확인.
