# 컬러 지시 게임(color-instruction) 구현 노트 (2026-07-16)

텐텐오락실 오마주 후보 10종(Phase 4) 중 여섯 번째이자 "동시 랭킹형" 계열의 마지막 게임.
구구단 스퍼트/홀짝 암산과 같은 서버 골격을 세 번째로 재사용했다 — 이 골격이 문제
생성·판정 로직만 바꿔서 재사용 가능하다는 계획서의 가정이 실제로 세 게임 연속 들어맞았다.

## 스펙에 없어 직접 정한 수치와 근거

- **4색(빨강/파랑/초록/노랑)**: 순서 기억 챌린지의 4타일과 같은 팔레트를 재사용해 이
  저장소 전체의 "색 게임"들이 시각적으로 통일감을 갖도록 했다.
- **문제 간격 2.2초**: 스트룹 효과(글자 뜻과 실제 색의 간섭)를 처리하려면 홀짝 암산의
  2.5초보다도 살짝 더 빠듯하게 잡아도 괜찮다고 판단 — 어차피 "뜻을 무시하고 색만 본다"는
  판단 자체는 사칙연산 암산보다 반사적으로 가능하다.
- **선택지는 텍스트 라벨 없는 색상 스와치 버튼**: 홀짝 암산의 "홀"/"짝" 버튼과 마찬가지로
  이진(사실은 4지선다) 판단 게임에 텍스트 입력은 과한 마찰이다. 버튼 자체에 색을
  칠해두면 "보이는 색을 그대로 탭"하는 게임 방식과 조작이 완전히 일치해서 설명 없이도
  직관적이다.

## 스트룹 효과 구현이 단순한 이유

`word`와 `displayColor`를 완전히 독립적으로 무작위 추첨했다 — 두 값이 우연히 같아지는
"일치 문제"(약 25%)도 자연스럽게 섞이는데, 이건 실제 스트룹 검사에서도 나타나는 정상적인
분포라 별도로 배제하지 않았다. 서버는 `displayColor`만 정답 판정에 쓰고 `word`는 클라이언트
렌더링(글자 자체)에만 쓰인다 — 두 값 다 브로드캐스트되는 공개 정보라 구구단 스퍼트·홀짝
암산과 동일하게 안티치트 이슈가 없다.

## 검증

- **raw WebSocket 스크립트**: 호스트가 서버가 보낸 `displayColor`를 그대로 제출해 5문제
  연속 정답 처리(`correctCount` 1→5)를 확인. 의도적으로 `word`(글자 뜻)를 제출해 오답
  처리가 정상 작동하는지도 확인 — 실제로 스트룹 간섭이 발생한 문제(`word !== displayColor`)
  에서 "뜻으로 답하면 틀린다"는 게임의 핵심 메커니즘이 서버에서 올바르게 판정됨을 검증했다.
- **Playwright UI 테스트**: 모바일 세로(390×844)에서 로비 → 카운트다운 → 플레이 화면까지
  실제 UI로 렌더링된 텍스트의 CSS `color` 값을 읽어 올바른 색 버튼을 탭하고, 정답 처리와
  보드 갱신을 스크린샷으로 확인.

## 배포

WAS 배포는 앞선 게임들과 동일한 절차(`ws-server/README.md`)를 따른다.
