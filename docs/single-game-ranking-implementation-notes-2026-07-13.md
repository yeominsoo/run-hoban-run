# 싱글 게임 랭킹/닉네임 기능 구현 노트 (2026-07-13)

## 요청 사항 (사용자 지시, v1/v2 계획서에는 없던 추가 요구)

- 모든 싱글 게임이 플레이가 끝났을 때 랭킹을 위한 닉네임을 입력받도록 한다.
- 이전에 입력한 이력이 있으면 그대로 선입력해도 된다.
- 랭킹 입력 후 게임별 대기실(=시작 화면)에서 랭킹보기 기능을 추가한다.

## 결정과 이유

1. **저장 범위: 기기 로컬(localStorage), 서버 없음**. 이 저장소의 싱글 게임들은 전부
   `src/shared/score-store.ts` 기반 로컬 최고점수 저장 구조라, "랭킹"도 같은 선상에서
   기기 로컬 상위 20개 기록으로 구현했다. 여러 사람이 같은 기기(키오스크)를 돌려가며
   플레이하는 행사 상황을 가정하면 "이 기기에서의 순위표"가 실제로 유용하다 — 별도
   서버·계정 시스템을 새로 만드는 건 이번 요청 범위를 크게 벗어난다고 판단했다(그렇게
   하려면 이 저장소에 처음으로 백엔드 데이터베이스가 필요해지는데, 지금 있는 유일한 서버는
   실시간 대전용 WebSocket 서버(`ws-server/`)이고 영속적 랭킹 저장소 역할을 하고 있지 않다).
2. **닉네임 재사용 범위: 게임 전체 공통 키 하나**. `rhh_last_nickname` 키 하나에 마지막으로
   쓴 닉네임을 저장해 모든 싱글 게임이 공유한다. 게임마다 닉네임을 다시 치게 하는 건
   불필요한 마찰이고, 한 사람이 여러 미니게임을 연달아 하는 행사 상황을 생각하면 게임을
   넘나들며 같은 이름을 기억해주는 게 자연스럽다.
3. **랭킹 개수 제한: 게임당 상위 20개**. 무제한으로 쌓으면 `localStorage` 용량과 렌더링
   비용이 계속 늘어난다. 20개면 소규모 행사에서 "역대 상위권"을 보여주기에 충분하다고
   판단했다(임의 수치지만, 스크롤 가능한 랭킹 리스트에서 한눈에 보기도 적당한 길이).
4. **저장 시점: 자동 저장이 아니라 "기록 저장" 버튼 클릭 시**. 게임이 끝나자마자 자동으로
   랭킹에 올리지 않고, 닉네임을 확인/수정할 기회를 준 다음 명시적으로 저장하게 했다.
   자동저장하면 오타난 이름이나 다른 사람이 실수로 만든 기록이 그대로 랭킹에 박제된다.
5. **버튼 재클릭 방지**: 저장 후 버튼을 비활성화해 같은 점수가 중복으로 여러 번 랭킹에
   쌓이는 것을 막는다. 재도전(다시 하기)하면 버튼이 다시 활성화된다.
6. **이름 검증**: 공백만 입력하거나 비워두면 "익명"으로 대체하고, 최대 12자로 자른다.
   XSS 방지를 위해 랭킹 리스트 렌더링 시 이름을 `escapeHtml()`로 이스케이프한다(다른
   사람이 이 기기에서 `<script>` 같은 문자열을 닉네임으로 넣고 저장할 가능성을 차단).
7. **"게임별 대기실에서 랭킹보기"의 해석**: 이 저장소의 싱글 게임들은 로비/대기실 개념이
   없고 "시작 전 안내 오버레이"가 그 역할을 한다. 그래서 시작 오버레이에 "랭킹보기" 버튼을
   추가하는 것으로 요청을 구현했다(멀티플레이 게임의 실제 "대기실"과는 별개 — 그쪽은
   과제 #13에서 다룬다).
8. **공용화 시점**: 이 기능을 붙이면서 그동안 게임마다 거의 동일하게 반복되던 헤더/오버레이/
   버튼 CSS(`.game-header`, `.overlay`, `.overlay-card`, `.primary-btn` 등)를
   `src/shared/game-shell.css`로 처음 통합했다. v2 계획서의 "게임 쉘 UI" 섹션에 적어둔
   원칙("공통 컴포넌트화는 3개 이상 게임에서 중복이 확인된 뒤 판단")대로, 이미 7개 게임이
   거의 동일한 블록을 복사해 쓰고 있어 공용화 조건을 충분히 넘겼다고 판단했다. JS 쪽 로직도
   `src/shared/leaderboard.ts`의 `setupRankingUI()`/`resetRankingSubmission()`로
   묶어서, 게임마다 반복될 뻔한 저장/조회 로직을 한 번만 구현했다.

## 추가 요청: 랭킹 이미지 캡처/공유 (사용자 지시)

랭킹 목록도 개인 결과 이미지처럼 캡처해서 저장/공유할 수 있어야 한다는 요청을 받아
`setupRankingUI()` 안에 통합했다.

- **결정**: 개인 결과 이미지(`aim-trainer`가 처음 도입한 패턴)와 동일한 스타일(어두운
  그라디언트 배경 + 카드형 행)로 랭킹 "목록 전체"를 캔버스에 그려 PNG로 만든다. 랭킹
  오버레이를 그대로 `html2canvas` 같은 라이브러리로 스크린샷하는 방식은 쓰지 않았다 —
  새 의존성을 추가해야 하고, DOM 캡처는 폰트/레이아웃이 기기마다 미묘하게 달라질 수
  있는 반면, 캔버스로 직접 그리면 완전히 통제된 고정 레이아웃의 공유용 이미지를 만들
  수 있다(이미 `dice`/`aim-trainer`에서 검증된 패턴을 재사용하는 것이기도 하다).
- **위치**: 개인 결과 이미지 버튼과 동일하게 "이미지 저장"(항상 노출, 다운로드)과
  "공유하기"(`navigator.share`+`canShare` 파일 지원 감지 시에만 노출) 두 버튼을
  랭킹 오버레이 안에 넣었다.
- **적용 범위**: `setupRankingUI()`가 이 버튼들의 존재를 전제로 하므로(필수 refs),
  랭킹 UI를 붙이는 모든 게임은 이 두 버튼도 함께 마크업에 추가해야 한다.

## 추가 요청: 게임 전 구간 상시 랭킹 버튼 (2026-07-22)

- 기기 내 랭킹을 쓰는 싱글 게임의 `랭킹보기` 버튼을 시작 오버레이에서 공용 헤더 우측으로
  옮겼다. 같은 DOM 버튼을 옮기는 방식이라 기존 이벤트와 접근성 id를 그대로 유지하면서
  시작 전, 플레이 중, 결과 화면 모두에서 랭킹을 열 수 있다.
- `setupRankingUI()`가 `.game-header`를 사용하는 게임에 공통 적용하므로 에임 트레이너,
  색맞추기 슬라이더, 볼 피하기, 타워 쌓기, 스네이크, 타이핑 서바이벌, 2048 변형,
  안엘런, 방치형 농장 9개 게임이 함께 반영된다.
- 주간 서버 랭킹 게임들은 이미 화면 전환과 무관한 상단 바에 랭킹 버튼이 상시 노출되어
  있다. 합이 10 퍼즐은 이 주간 랭킹 버튼을 유지해 상단에 로컬/서버 버튼이 중복되지 않게 했다.

## 추가 요청: 다른 기기 사용자 점수 통합 (2026-07-23)

- 기존 `localStorage` 랭킹만 열던 싱글게임 10종을 WAS의 전체 점수 랭킹
  `GET/POST /ranking/score/<game>`에 연결했다. 랭킹 창에는 닉네임별 최고 점수를 내림차순으로
  최대 50명까지 표시한다.
- 업데이트 전 기기에 남아 있는 로컬 상위 20개 기록은 해당 기기에서 게임 페이지를 처음 열 때
  서버로 전송해 합친다. 서버에는 과거 로컬 기록의 사본이 없으므로 업데이트 후 다시 접속하지
  않는 기기의 옛 기록까지 자동 복구할 수는 없다.
- 서버 전송이 실패하면 랭킹 창은 종전처럼 이 기기의 로컬 기록을 표시한다. 로컬 저장도 계속
  유지하므로 일시적인 WAS 장애 때문에 방금 저장한 기록이 사라지지 않는다.
- 같은 닉네임은 최고 점수 하나만 유지하고, 낮은 점수를 다시 제출해도 기존 최고점을 덮어쓰지
  않는다. 기존 로컬 목록이나 서버 응답에 동일 닉네임이 여러 건 있어도 화면 표시·이미지 내보내기
  전에 최고점 한 건으로 합친다. 서버 데이터는 기존 승/패 랭킹과 분리된
  `score-ranking-<game>.json` 파일에 저장한다.

## 추가 요청: 자동 기록 + 게임 통합 랭킹 (2026-07-25)

- **저장 버튼 없이 자동 기록**: 닉네임이 이미 기기에 저장돼 있으면(=한 번이라도 저장 폼에
  입력한 적이 있으면), 다음 판부터는 "기록 저장" 버튼을 누르지 않아도 게임이 끝날 때 자동으로
  기록한다 — 단, 이번 점수가 그 닉네임의 기존 최고 기록보다 **높을 때만**. 닉네임이 아직 없는
  최초 플레이는 여전히 수동 입력 폼을 그대로 쓴다(오타·장난 닉네임이 자동으로 박제되는 것을
  막기 위해 "닉네임 확인 없이 저장하지 않는다"는 2026-07-13 결정 4번 원칙은 유지하되, 이미
  확인된 닉네임에 한해서만 자동화했다). `setupRankingUI()`가 반환하는 `reset()` 클로저가 이
  비교·자동 저장을 수행하며, 각 게임은 결과 화면을 다시 보여줄 때(또는 `idle-farm`처럼 점수가
  오를 때마다) 이 함수를 호출하기만 하면 된다. 저장 버튼은 계속 활성 상태로 남겨 닉네임을
  바꿔 별도로 저장하고 싶은 경우에도 쓸 수 있다.
- **10개 게임 통합 랭킹**: "랭킹보기"를 열면 이제 이 게임 하나가 아니라 점수 랭킹을 지원하는
  10개 싱글게임 전체를 점수 내림차순으로 합친 목록을 보여준다. 같은 닉네임이 여러 게임에서
  기록을 냈다면 게임마다 한 줄씩 따로 나타난다(점수 단위가 게임마다 달라 하나의 숫자로 합산하는
  건 의미가 없고, "어떤 게임에서 낸 기록인지"가 더 중요하다고 판단했다). 각 줄에 게임 이름을
  작게 붙여 구분한다(`.ranking-game`). 서버는 새 예약 엔드포인트 `GET /ranking/score/_all`로
  게임별 저장소(`score-ranking-<game>.json`) 10개를 즉석에서 합쳐 반환한다 — 통합용 파일을
  별도로 두지 않아 기존 게임별 저장/집계 로직과 데이터를 그대로 재사용한다. 게임별 저장/조회
  엔드포인트(`GET/POST /ranking/score/<game>`)는 그대로 남아 있고, 각 게임은 여전히 자기
  엔드포인트로만 점수를 올린다 — `_all`은 조회 전용이며 POST를 받지 않는다(405).
