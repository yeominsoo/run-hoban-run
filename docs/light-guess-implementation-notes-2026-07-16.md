# 라이트 게스(light-guess) 구현 노트 (2026-07-16)

텐텐오락실 오마주 후보 10종의 마지막 게임. Phase 5 "실시간 조작 동기화형"이지만 줄다리기
배틀·영역 쟁탈전과 달리 **다인(2~8명) + 다회 라운드 서바이벌**이라는 점에서 오히려
순서 기억 챌린지(memory-sequence)의 로비/라운드/타이브레이크 골격에 더 가깝다 — Phase
구분(4 vs 5)이 "판정 방식"(문제 배포형 vs 실시간 동기화형) 기준이지 "인원 수" 기준이
아니라는 걸 이 게임에서 체감했다.

## 스펙 문구를 실제 규칙으로 구체화한 과정

계획서 한 줄 명세("매 라운드 무작위로 켜지는 조명 앞에서 멈추면 탈락, 마지막 생존자
승리")는 두 가지로 해석 가능했다: (1) 정통 "무궁화꽃이피었습니다"(초록불=이동,
빨간불=정지, 못 멈추면 탈락) 또는 (2) 문구를 글자 그대로 읽어 "멈추는 행위 자체가
탈락 조건"(그러면 아무것도 안 하는 게 최선의 전략이 되어버려 게임이 성립하지 않는다).
두 번째 해석은 실제로 플레이 가능한 게임이 되지 않아서 기각하고, 아래처럼 정통 규칙에
"멈추면 탈락"이라는 표현이 정확히 들어맞도록 조합했다.

- **초록불(2~4.5초 무작위) 동안 최소 1회 이상 탭해야 한다.** 이 라운드에 한 번도
  탭하지 않은 사람은 "멈춰 있었다"고 판정돼 빨간불이 켜지는 순간 탈락한다 — 이 부분이
  스펙의 "멈추면 탈락"을 문자 그대로 구현한 지점이다.
  - "계속 눌러야 한다(초당 N회 이상)"는 요구도 고려했지만, 모바일에서 손가락이
    피로해지는 마찰이 너무 커서 "라운드당 최소 1회"로 완화했다 — 그래도 완전히 손을
    놓고 있으면 걸리므로 "계속 지켜봐야 한다"는 긴장감은 유지된다.
- **빨간불(1.2초 고정) 동안 탭하면 즉시 탈락.** 반대로 "정지 신호에 움직이면 걸린다"는
  RLGL의 핵심 긴장감이다.
- **언제 빨간불로 바뀌는지는 예측 불가능해야 한다** — 서버만 알고 있다가 실제로
  전환되는 순간 브로드캐스트한다(두더지 사냥의 무작위 스폰 딜레이와 같은 설계).

## 두 가지 탈락 조건을 하나의 원칙으로 통합

"초록불에 안 움직여서 탈락"과 "빨간불에 움직여서 탈락"은 서로 반대되는 조건처럼
보이지만, "신호를 어기면 탈락"이라는 한 가지 규칙의 두 얼굴이다. 서버 구현에서도
`hasTappedThisGreen`(초록불 동안의 최소 참여 여부)과 `alive`에서의 즉시 제거(빨간불
탭 감지)라는 서로 다른 체크 지점만 있을 뿐, 둘 다 결국 `alive` 집합에서 토큰을
제거하고 `eliminatedAt` 타임스탬프를 기록하는 동일한 마무리 로직으로 수렴한다.

## 검증

- **raw WebSocket 스크립트 (시나리오 1)**: 호스트만 초록불 내내 탭하고 게스트는 전혀
  탭하지 않아, 빨간불 전환 시 게스트가 "멈췄다"로 정확히 탈락 처리되고 즉시
  `game_over`(호스트 우승)가 오는지 확인.
- **raw WebSocket 스크립트 (시나리오 2)**: 양쪽 다 초록불 내내 탭해 아무도 "정지"로
  탈락하지 않음을 확인한 뒤, 빨간불 전환 이후에도 계속 탭을 흘려보내 그중 한쪽이
  실제로 `caught_moving`으로 탈락하고 생존자가 즉시 우승 처리됨을 확인.
- **Playwright UI 테스트**: 모바일 세로(390×844)에서 로비 → 카운트다운 → 초록불(초록
  구슬 + "계속 탭하세요!") → 빨간불(빨간 구슬 + "손을 멈추세요!") 전환이 실제 화면에서
  시각적으로 확인됐고, 두 탭이 계속 탭을 흘려보내는 도중 빨간불 전환 타이밍에 걸린
  쪽이 실시간으로 탈락 처리되어 최종 결과 화면까지 자연스럽게 넘어가는 전체 흐름을
  확인했다.

## 배포

WAS 배포는 앞선 게임들과 동일한 절차(`ws-server/README.md`)를 따른다. 이 게임을
끝으로 텐텐오락실 오마주 후보 10종(Phase 4~5)이 전부 완료됐다.
