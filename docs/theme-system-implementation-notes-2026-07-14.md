# 테마 시스템 구현 노트 (2026-07-14)

## 요청 원문

> 카지노 분위기의 웹페이지로 전체적으로 변경이 필요한데 지금 테마도 마음에 들어서 아예
> 메인페이지에서 테마기능도 추가하자. 카지노 / 소녀감성 / 사이버펑크 / 구름구름해(하늘색과
> 흰색 조합의 밝은느낌) 설정버튼 같은걸 제공해서 테마 선택하게 해줘

바로 앞서 나온 "방치형 농장/무한 러너/스네이크 그래픽이 밋밋하다"는 피드백은 결국 이
요청으로 이어졌다고 판단했다 — 개별 게임 그래픽을 손보는 대신, 사이트 전체의 분위기를
바꿀 수 있는 테마 시스템을 요청한 것으로 이해하고 이 방향으로 진행했다.

## 결정과 이유

1. **`<html data-theme="...">` + CSS 커스텀 프로퍼티 오버라이드 방식.** 다크모드 토글의
   표준 패턴과 동일하다. 이미 이 저장소의 모든 게임 CSS가 `shared/pastel-theme.css`의
   `--color-*` 변수를 참조하고 있어서(레이스/팀/주사위/가위바위보/라이어/마피아/할리갈리/
   윷놀이/전략윷놀이까지 전부 이 파일을 `@import`한다), 이 파일 하나에 테마별
   `:root[data-theme="X"] { --color-*: ... }` 블록만 추가하면 **19개 게임 페이지 전부에
   자동으로 전파**된다 — 페이지마다 별도 테마 로직을 넣을 필요가 없다.
2. **테마 선택 버튼은 요청대로 허브(메인페이지)에만 배치.** 게임 페이지에는 선택 UI가
   없고, 저장된 테마를 읽어 적용만 한다. 게임 진행 중 테마를 바꾸는 시나리오 자체가
   없으므로, 캔버스 게임들이 `getComputedStyle()`로 색상을 페이지 로드 시점에 1회만
   읽어와 상수로 고정해두는 기존 패턴(예: 에임 트레이너의 `COLOR_PRIMARY`)과도 충돌이
   없다 — 게임 페이지에 처음 진입하는 순간에는 이미 테마가 적용되어 있으니 정확한 색을
   읽어온다.
3. **FOUC(테마 적용 전 원래 색이 잠깐 보이는 깜빡임) 방지를 위해 20개(허브+19게임)
   `index.html`의 `<head>` 맨 앞에 동일한 인라인 스크립트를 삽입.** Vite가 CSS/JS를
   `<script type="module">`로 지연 로드하므로, `main.ts`가 실행되고 나서야
   `data-theme`를 설정하면 아주 짧게라도 원래 테마로 렌더링되는 깜빡임이 생긴다. HTML이
   파싱되자마자(다른 무엇보다 먼저) 실행되는 인라인 스크립트로 `localStorage`를 읽어
   `document.documentElement.dataset.theme`를 즉시 세팅해 이 문제를 없앴다.
   `shared/theme.ts`의 `applyStoredTheme()`도 동일 로직을 제공하지만, 이건 인라인
   스크립트가 없는 테스트 환경 등을 위한 보조 수단이고 실제 프로덕션 경로는 인라인
   스크립트다.
4. **캔버스/DOM 셸까지만 테마 적용, 3D 씬(레이스의 Three.js 배경 등)은 이번 스코프에서
   제외.** `src/main.ts`의 `scene.background = new THREE.Color(0xb8d9ff)`처럼 3D
   렌더링 배경/조명 색상은 CSS 변수가 아니라 JS 상수로 하드코딩되어 있다. 이걸 테마별로
   바꾸려면 3D 씬 자체를 재설계해야 해서 스코프가 크게 벌어진다 — 사용자가 원한 "사이트
   전체 분위기"는 헤더·버튼·카드·오버레이 등 UI 셸 색상으로 충분히 달성된다고 판단해,
   3D 게임 월드 내부(레이스 트랙, 윷놀이 보드 등) 렌더링 색상은 그대로 두기로 했다. 각
   게임의 캔버스 2D 렌더링(에임 트레이너의 원, 스네이크의 몸통 등)은 `--color-*` 변수를
   읽어 쓰므로 자동으로 테마를 따라간다.
5. **공용 셸(`game-shell.css`)과 허브(`hub.css`)에 하드코딩되어 있던 `rgba(255,255,255,
   X)` 반투명 흰색들을 전부 테마 변수로 교체.** 처음엔 `--color-surface*` 변수만 있으면
   충분할 줄 알았는데, 실제로 헤더·오버레이·버튼·랭킹 목록 등 곳곳에 반투명 흰색이
   그대로 박혀 있어서, 다크 테마(카지노/사이버펑크)에서는 밝은 반투명 막이 어두운
   배경 위에 뜬 것처럼 보여 텍스트 대비가 깨지는 문제가 있었다. `--color-surface-weak`
   /`--color-surface-hover`/`--color-overlay-scrim`/`--color-card-gradient`/
   `--color-card-gradient-hover`/`--color-icon-gradient`/`--color-icon-inset-highlight`
   7개 변수를 새로 추가해 이 하드코딩들을 대체했다. **게임별로 개별 하드코딩된 이펙트
   색상(스네이크의 5색 순환 테마, 각 게임 히트 플래시 색 등)은 게임 고유의 연출로 보고
   이번 스코프에서 건드리지 않았다** — 이런 것까지 전부 손보면 19개 게임을 다시 하나씩
   감사해야 해서 스코프가 지나치게 커진다.
6. **4개 팔레트 설계**:
   - **구름구름해**: 기존 파스텔 테마 그대로(`:root` 기본값). 사용자가 "지금 테마도
     마음에 든다"고 했으므로 그대로 유지하고 이름만 붙였다.
   - **카지노**: 어두운 그린 펠트 배경(`#0a2118` 계열) + 골드(`#c9a227`) 포인트. 카드
     테이블의 초록 펠트와 칩의 금색이라는 가장 직관적인 "카지노" 연상을 그대로 썼다.
   - **소녀감성**: 밝은 핑크(`#ff5ca8`) + 라벤더(`#c9a8ff`) 배경. 기존 구름구름해와
     구분되도록 배경 자체를 핑크/라벤더 그라데이션으로 더 진하게, 포인트 컬러도 더
     채도 높은 핫핑크로 잡았다.
   - **사이버펑크**: 거의 검정에 가까운 네이비(`#0a0e27`) 배경 + 네온 마젠타
     (`#f724c9`) + 네온 시안(`#00e5f0`). 장르 관습적인 네온 대비 배색.
7. **테마 선택 UI에 실제 CSS 변수 대신 하드코딩된 미리보기 색을 썼다.** 선택 목록의
   스와치(원형 그라데이션)는 `getComputedStyle`로 각 테마의 실제 변수를 읽어오는 대신,
   `THEME_SWATCHES` 상수에 미리보기용 색을 직접 적어뒀다 — 그래야 "지금 적용된 테마와
   무관하게" 4개 스와치가 각자의 실제 색으로 항상 보인다(만약 CSS 변수를 참조했다면 현재
   테마가 아닌 다른 3개 테마의 스와치를 그릴 방법이 없다, 그 테마가 활성화되어 있지 않은
   한 해당 변수 값을 읽을 수 없으므로).

## 파일 변경 요약

- `src/shared/pastel-theme.css`: 4개 테마 블록(`:root`, `[data-theme="casino"]`,
  `[data-theme="girlish"]`, `[data-theme="cyberpunk"]`), 변수 33개 × 4테마.
- `src/shared/theme.ts` (신규): `ThemeId`, `THEMES` 목록, `loadTheme`/`saveTheme`/
  `applyStoredTheme`.
- `src/shared/game-shell.css`, `src/pages/hub/hub.css`: 하드코딩 흰색 → 테마 변수.
- `src/pages/hub/main.ts`, `hub.css`: 헤더에 테마 설정 버튼 + 테마 선택 오버레이 추가.
- 20개 `index.html`(허브 1 + 게임 19): FOUC 방지 인라인 스크립트 삽입.

## 테스트 전략

허브에서 테마를 선택하면 `localStorage`에 저장되고 `<html data-theme>`가 즉시 바뀌는지,
그리고 다른 페이지로 이동해도(재방문 시 인라인 스크립트가 먼저 실행되어) 유지되는지를
검증한다. 3D 씬 색상은 스코프에서 제외했으므로 테스트 대상이 아니다.
