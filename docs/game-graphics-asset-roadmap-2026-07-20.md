# 게임 그래픽 에셋 개선 로드맵

- 작성일: 2026-07-20
- 대상: Toris Arcade 허브에 등록된 30종 게임
- 목적: 플레이 핵심 오브젝트가 단순 도형·이모지에 머문 게임을 선별하고, 저작권과 성능을 통제한
  전용 그래픽 에셋으로 순차 개선한다.

## 1. 선정 근거

2026-07-20 기준 소스와 로컬 렌더 화면을 함께 확인했다. 아래 조건을 많이 만족할수록 우선순위를
높였다.

1. 플레이 중 계속 바라보거나 직접 눌러야 하는 핵심 오브젝트가 도형 또는 운영체제 이모지다.
2. 그래픽을 바꿔도 서버 판정, 충돌 범위, 점수 규칙은 유지할 수 있다.
3. 작은 모바일 화면에서도 에셋이 상태 판독을 더 분명하게 만든다.
4. 한 게임에 필요한 에셋 수와 초기 다운로드 용량을 통제할 수 있다.
5. Toris Arcade 고유의 밝고 친근한 마스코트 스타일을 만들 여지가 크다.

### 개선 대상과 순서

| 순서 | 게임 | 현재 한계 | 도입할 핵심 에셋 | 확보 방식 | 우선순위 |
|---:|---|---|---|---|---|
| 1 | 볼 피하기 + 수집 | 플레이어·위험구·수집구가 광택 원으로만 표시됨 | 플레이어 마스코트, 위험 운석, 수집 별 | 신규 이미지 제작 | P0 |
| 2 | 두더지 사냥 | 두더지가 시스템 이모지라 OS에 따라 모양이 달라지거나 깨짐 | 전용 두더지 캐릭터 | 신규 이미지 제작 | P0 |
| 3 | 할리갈리 | 과일·종·카드 뒷면이 시스템 이모지/문자에 의존 | 과일 4종, 종, 카드 뒷면 | 라이선스가 명확한 무료 에셋 + 자체 카드 프레임 | P0 |
| 4 | 줄다리기 배틀 | 핵심 장면이 진행 바와 큰 버튼만으로 구성됨 | 양 팀 마스코트, 줄·깃발이 있는 경기장 장면 | 신규 이미지 제작 | P0 |
| 5 | 라이트 게스 | 신호등 원과 탭 버튼만 있어 생존 경주의 장면성이 약함 | 달리는 참가자와 운동장 배경 | 신규 이미지 제작 | P1 |
| 6 | 방치형 농장 | 빈 밭과 작물 상태가 CSS 아이콘 중심이고 배경 공간이 비어 있음 | 농장 배경, 밭 장식 레이어 | 신규 이미지 제작 | P1 |

### 이번 대상에서 제외한 게임

- `race`, `endless-runner`: GLB 및 전용 캐릭터 애니메이션 에셋 체계가 이미 있다. 신규 에셋 추가보다
  기존 에셋 최적화와 회귀 검증이 우선이다.
- `aim-trainer`, `color-slider`, `tower-stack`, `snake`, `2048-hex`, `memory-sequence`,
  `updown-number`, `multiplication-sprint`, `odd-even-math`, `color-instruction`,
  `sum-ten-puzzle`, `territory-clash`, `reversi`, `gomoku`: 색·숫자·격자·보드의 즉각적인 판독이
  핵심이다. 장식 에셋을 늘리면 플레이 정보보다 시각 잡음이 커질 가능성이 높다.
- `team`, `dice`, `rps`, `liar`, `mafia`, `yutnori`, `strategy-yutnori`: 현재 고유한 결과 연출,
  손 아이콘, 역할 카드 또는 3D 오브젝트가 있으며 이번 6종보다 에셋 교체 효과가 낮다.
- `typing-survival`: 단어 판독과 입력 지연이 핵심이므로 배경 에셋 추가를 보류한다.

제외는 영구 금지가 아니다. 이번 6종을 모두 배포하고 전체 회귀 검증을 끝낸 뒤 실제 사용자 피드백이
있을 때 별도 작업으로 재평가한다.

## 2. 공통 아트 디렉션

### 시각 언어

- 스타일: 둥근 실루엣의 2D 스티커 일러스트, 두꺼운 외곽선, 단순한 명암 2~3단계
- 분위기: 밝고 장난스러운 행사장/학교 축제 느낌
- 기본 팔레트: 코랄 `#ff5c7f`, 민트 `#59cfbf`, 골드 `#f5b83d`, 잉크 `#3f3140`
- 배경: 기존 테마 색을 가리지 않는 저채도·저대비
- 금지: 실제 브랜드 로고, 저작권 캐릭터, 읽을 수 없는 생성형 문자, 과도한 사실 묘사, 공포 표현

### 제작 규격

| 용도 | 원본 권장 크기 | 런타임 형식 | 권장 용량 |
|---|---:|---|---:|
| 플레이 오브젝트 | 512×512 이상 정사각형 | 투명 PNG 또는 WebP | 개별 180KB 이하 |
| 카드/아이콘 | 256×256 이상 | SVG 또는 투명 WebP | 개별 60KB 이하 |
| 와이드 배경 | 1536×864 이상 | WebP | 개별 350KB 이하 |
| 모바일 장식 레이어 | 768×1024 이상 | WebP | 개별 250KB 이하 |

- CSS 표시 크기보다 2배 이상 큰 원본을 사용해 고밀도 화면에서 흐려지지 않게 한다.
- 장식 배경은 `image-set()` 또는 WebP 한 장으로 제공하고, 중요한 상태 표현은 이미지 색에만
  의존하지 않는다.
- 플레이 오브젝트의 충돌 영역과 버튼 영역은 기존 DOM/Canvas 수치를 유지한다.
- 에셋 로딩 실패 시에도 게임이 가능하도록 CSS 도형 또는 텍스트 대체 경로를 남긴다.
- 시작 오버레이가 열려 있는 동안 필요한 이미지를 미리 로딩한다.

## 3. 에셋 출처와 권리 관리

### 외부 무료 에셋

1. 상업적 이용과 수정이 허용되는 라이선스만 사용한다.
2. 원본 URL, 제작자/프로젝트, 라이선스 이름, 내려받은 날짜를 게임별 `LICENSE.md`에 기록한다.
3. 원격 URL을 직접 참조하지 않고 저장소 안에 복사해 빌드 산출물에 포함한다.
4. 라이선스 파일과 고지 문구가 요구되면 에셋과 함께 커밋한다.
5. 출처 또는 라이선스가 불명확하면 사용하지 않는다.

### 신규 제작 이미지

1. 프롬프트에는 게임명보다 피사체·동작·색·구도·배경 제거 조건을 먼저 적는다.
2. 생성 이미지 안에 문자, 숫자, 로고를 넣지 않는다.
3. 투명 오브젝트는 단색 크로마키 배경으로 생성한 뒤 알파 PNG로 변환하고 가장자리를 검사한다.
4. 최종 채택 파일과 프롬프트, 생성 도구/모델, 생성일을 게임별 `ASSET-MANIFEST.md`에 기록한다.
5. 채택하지 않은 시안은 배포 경로에 넣지 않는다.

## 4. 게임별 개선 방향과 완료 기준

### 4.1 볼 피하기 + 수집

- 플레이어: 화면 중앙을 향하는 민트색 비행 토끼/별 조종사 마스코트
- 위험구: 코랄색 가시 운석. 붉은 계열과 뾰족한 실루엣으로 위험을 이중 표현
- 수집구: 골드/민트 별. 색뿐 아니라 별 실루엣으로 보상을 구분
- 적용: Canvas `drawImage()`를 사용하되 기존 원형 충돌 반경은 변경하지 않는다.
- 완료 기준: 이미지 로딩 성공/실패 양쪽에서 시작·드래그·충돌·무적 깜빡임·게임오버가 동작하고,
  모바일에서 세 오브젝트가 즉시 구분된다.

### 4.2 두더지 사냥

- 정면을 보는 갈색 두더지 캐릭터 한 종을 투명 PNG로 제작한다.
- `active`, `hit`, `miss` 상태는 같은 이미지에 CSS 이동·회전·채도 효과를 적용해 용량을 줄인다.
- 구멍은 현재 CSS 도형을 유지해 타격 영역과 레이아웃을 보존한다.
- 완료 기준: 시스템 이모지 의존이 없어지고, 9개 홀의 활성/명중/놓침 애니메이션과 클릭 판정이
  기존 테스트 조건을 유지한다.

### 4.3 할리갈리

- 과일 4종과 종은 라이선스가 명확한 무료 벡터 에셋을 사용한다.
- 카드 프레임과 뒷면 패턴은 CSS로 유지하되, 과일 개수 배치는 이미지 요소로 렌더한다.
- 숫자 카운트는 보조 정보로 남겨 색각·저시력 사용자가 개수를 다시 확인할 수 있게 한다.
- 완료 기준: 이모지가 지원되지 않는 브라우저에서도 과일·종이 보이며, 1~5개 배치, 카드 뒤집기,
  올바른/잘못된 종 치기 상태가 유지된다.

### 4.4 줄다리기 배틀

- 왼쪽 민트 팀과 오른쪽 코랄 팀의 동물 마스코트를 하나의 와이드 장면으로 제작한다.
- 줄 중앙 깃발 위치는 기존 진행 값과 연결된 별도 DOM 요소로 유지한다.
- 배경은 장식이며 이름·시간·진행 바·당기기 버튼 대비를 침범하지 않는다.
- 완료 기준: 진행 값 -1~1에서 깃발이 끝점 밖으로 나가지 않고, 390px 모바일에서도 양 팀과
  현재 우세 방향이 보인다.

### 4.5 라이트 게스

- 운동장 트랙과 출발선, 작은 참가자 실루엣을 포함한 저대비 배경을 제작한다.
- 빨강/초록 신호는 현재 CSS 전구와 텍스트를 유지한다. 배경 이미지가 판정 신호를 대신하지 않는다.
- 탈락자는 참가자 칩의 기존 상태로 표시하고, 배경 캐릭터는 장식으로만 사용한다.
- 완료 기준: 빨강/초록 신호 대비가 WCAG 비텍스트 대비 취지에 맞게 유지되고, 배경 로딩 실패 시에도
  탭/정지 판단에 영향이 없다.

### 4.6 방치형 농장

- 밭·울타리·헛간·하늘을 담은 저대비 농장 배경을 제작한다.
- 6개 밭 버튼과 작물 성장 단계는 기존 CSS/DOM 상태를 유지해 오프라인 시간 계산과 자동 수확
  로직에 영향을 주지 않는다.
- 배경은 데스크톱과 모바일에서 중앙 밭 영역을 가리지 않도록 `cover` 크롭 안전 영역을 둔다.
- 완료 기준: 시작 오버레이·작물 선택·성장·수확·업그레이드·랭킹 입력의 텍스트 대비가 유지되고,
  390px 화면에서 가로 스크롤이 생기지 않는다.

## 5. 구현·검증·배포 절차

각 게임을 하나의 독립 케이스로 처리한다.

1. 현재 화면을 데스크톱 1280×900, 모바일 390×844로 캡처한다.
2. 에셋을 확보 또는 제작하고 게임별 매니페스트/라이선스를 기록한다.
3. 해당 게임의 코드·스타일·테스트만 수정한다.
4. `npm run build`와 해당 게임 Playwright 테스트를 실행한다.
5. 이미지 크기, 알파 채널, 투명 모서리, 실제 네트워크 요청 성공 여부를 검사한다.
6. 변경 후 같은 화면을 다시 캡처해 오브젝트 판독·텍스트 대비·오버플로를 비교한다.
7. 게임 단위 커밋을 만들고 `master`에 푸시한다.
8. `npm run deploy:firebase`로 배포하고 배포 마커가 `origin/master`에 반영됐는지 확인한다.
9. WebSocket 프로토콜/서버 로직을 바꾸지 않은 그래픽 케이스는 WAS를 재배포하지 않는다.
